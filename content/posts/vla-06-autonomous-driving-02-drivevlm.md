# DriveVLM 详解

> ⚙️ 进阶 | 前置知识：[端到端驾驶 VLA 范式](./01-vla-driving-paradigm.md)、[VLM 主干网络](../07-foundation-models/02-vlm-backbone.md)

## 概述

**DriveVLM**（Tian et al., 2024）是驾驶 VLA 领域的代表性工作，首次系统性地将大型视觉-语言模型（VLM）的链式推理能力应用于自动驾驶决策。其核心贡献在于设计了**三阶段思维链（Chain-of-Thought, CoT）** 推理框架——场景描述、场景分析、层级规划——使驾驶决策过程可用自然语言表达和追溯。

论文同时提出 **DriveVLM-Dual** 架构，将 VLM 的语义推理与传统几何规划器融合，在长尾场景处理和精确轨迹生成之间取得平衡。

## 🔰 VLM 主干：模型选择与输入输出格式

### 主干网络

DriveVLM 基于 **InternVL**（Chen et al., 2024）系列 VLM 构建，具体使用 InternVL-Chat 变体：

| 组件 | 规格 | 说明 |
|------|------|------|
| 视觉编码器 | InternViT-6B | 基于 ViT-6B，输入分辨率 448×448 |
| 语言模型 | InternLM2-20B | 20B 参数 LLM，支持中英文 |
| 视觉-语言适配 | Dynamic Pixel Shuffle | 将视觉 token 压缩后送入 LLM |
| 总参数量 | ~26B | 全模型端到端训练 |

### 输入格式

```
视觉输入: 6 路环视 (前/前左/前右/后/后左/后右)
         每张 1600×900 → resize → 448×448
         时序: 当前帧 + 过去 3 帧 (共 24 张图像)

文本输入 (System Prompt):
  "你是一个自动驾驶系统。请根据环视图像进行场景描述、
   场景分析和层级规划。输出格式:
   <scene_desc>...</scene_desc>
   <scene_analysis>...</scene_analysis>
   <planning>...</planning>"
```

### 输出格式

模型以结构化 XML 标签输出，每个标签对应 CoT 的一个阶段：

```xml
<scene_desc>
  前方 50m 处有一辆白色卡车正在从右车道向左车道变道，
  速度约 30km/h。右侧车道有一辆公交车以 20km/h 慢速行驶。
  本车当前速度 60km/h，位于中间车道。天气晴朗，路面干燥。
</scene_desc>

<scene_analysis>
  关键物体: [卡车, 优先级=高, 距离=50m, 动作=变道]
           [公交车, 优先级=中, 距离=35m, 动作=直行]
  风险评估: 中高风险。卡车变道可能在 2s 内切入本车道。
  约束条件: 右侧超车被公交车阻挡，左侧车道需确认安全。
</scene_analysis>

<planning>
  meta_action: DECELERATE_AND_KEEP_LANE
  decision: 保持当前车道，匀减速至 35km/h，等待卡车完成变道
  speed_plan: [60, 52, 45, 38, 35] km/h  (per 0.5s)
  trajectory: [(0.0, 0.0), (0.2, 8.1), (0.3, 15.0),
               (0.2, 20.8), (0.1, 25.5), (0.0, 29.2)]
</planning>
```

## ⚙️ 三阶段 CoT 推理：深度解析

### Stage 1: 场景描述（Scene Description）

**目标**：将视觉感知结果转化为结构化的自然语言描述。

**设计动机**：传统感知模块输出检测框和类别标签，丢失了大量上下文信息（如"卡车正在变道"只能从连续帧的框位移间接推断）。VLM 可以直接从图像"看到"并"说出"动态行为。

**描述维度**：
- 交通参与者：类型、位置、速度估计、行为描述
- 道路环境：车道数、路口类型、施工/事故
- 天气/光照：影响感知和操控的环境因素
- 交通信号：红绿灯状态、交通标志内容

### Stage 2: 场景分析（Scene Analysis）

**目标**：基于场景描述进行风险评估和关键实体优先级排序。

**设计动机**：这是 VLA 相比纯视觉端到端的核心差异化——VLM 的常识推理能力可以处理训练数据中未出现的场景。例如，模型从未见过"倒在路中间的梯子"，但可以推理出这是障碍物并需要绕行。

**分析框架**：
```
关键实体排序:
  [实体1, 优先级=高/中/低, 原因="..."]
  [实体2, ...]

风险等级: 低/中/中高/高/极高
风险来源: "..."

行为预测:
  [实体1 可能的行为: "..."]
  [时间窗口: Ns]
```

### Stage 3: 层级规划（Hierarchical Planning）

**目标**：从高层决策到低层轨迹的层级生成。

**三层规划结构**：

| 层级 | 内容 | 输出形式 | 示例 |
|------|------|---------|------|
| 高层 (Meta Action) | 宏观驾驶行为 | 枚举类别 | LANE_CHANGE_LEFT, DECELERATE |
| 中层 (Speed Plan) | 速度曲线 | 数值序列 | [60, 50, 40, 35] km/h |
| 低层 (Trajectory) | 2D 轨迹点 | (x, y) 坐标序列 | [(0,0), (1.2, 8.5), ...] |

**层级规划的好处**：即使低层轨迹有偏差，高层决策的正确性仍可保证安全方向；同时可解释性集中在高层决策上。

## ⚙️ 三个完整 CoT 推理示例

### 示例 1：高速公路突发障碍物

```
[输入] 6 路环视图像: 高速公路场景，前方车道中央有掉落的轮胎碎片

<scene_desc>
  当前位于三车道高速公路中间车道，速度 100km/h。
  前方约 80m 处中间车道路面上有黑色轮胎碎片，占据约半个车道宽度。
  左车道有一辆轿车距本车约 40m，速度相近。右车道空闲。
  后方无紧跟车辆。能见度良好。
</scene_desc>

<scene_analysis>
  关键物体: [轮胎碎片, 优先级=极高, 距离=80m, 类型=静态障碍物]
           [左侧轿车, 优先级=高, 距离=40m, 动作=直行, 阻挡左变道]
  风险评估: 高风险。以当前速度 3s 内到达障碍物。
  约束: 左变道被邻车阻挡(不安全)。右变道可行(车道空闲)。
  时间预算: 约 2.8s 完成避障。
</scene_analysis>

<planning>
  meta_action: LANE_CHANGE_RIGHT
  decision: 右变道避开障碍物，同时适度减速增加反应时间
  speed_plan: [100, 92, 85, 80, 80] km/h
  trajectory: [(0.0, 0.0), (-0.8, 13.5), (-2.1, 25.8),
               (-3.5, 36.0), (-3.7, 44.8), (-3.7, 52.5)]
</planning>
```

### 示例 2：城市交叉路口行人意图推理

```
[输入] 6 路环视图像: 城市路口，绿灯直行

<scene_desc>
  无保护左转路口，本车直行方向绿灯。速度 40km/h。
  右前方人行横道边缘站着一位老人，面朝马路方向，
  手持拐杖，身体前倾，似有横穿意图。
  对向车道有 2 辆车正在通过路口。
</scene_desc>

<scene_analysis>
  关键物体: [老年行人, 优先级=极高, 距离=25m, 意图=可能横穿]
  风险评估: 高风险。老年行人移动缓慢，横穿时间长。
  常识推理: 老人身体前倾+面朝马路 = 高概率即将横穿。
           即使本车绿灯，也应优先保护弱势道路使用者。
  约束: 紧急刹车距离约 15m(40km/h, 干燥路面)。
</scene_analysis>

<planning>
  meta_action: DECELERATE_AND_YIELD
  decision: 预防性减速至 15km/h，准备让行。若行人开始横穿则停车。
  speed_plan: [40, 32, 24, 18, 15] km/h
  trajectory: [(0.0, 0.0), (0.0, 5.0), (0.0, 9.2),
               (0.0, 12.5), (0.0, 15.0)]
</planning>
```

## ⚙️ DriveVLM-Dual 架构

### 设计动机

单独的 VLM 推理存在两个缺陷：(1) 生成延迟 300-800ms，无法满足实时性；(2) 自回归生成的轨迹坐标精度有限。DriveVLM-Dual 通过并行运行 VLM 推理与传统规划器来弥补。

### 架构图

```
多视角环视图像 (6 cameras × 4 frames)
     │
     ├──────────────────────┬──────────────────────┐
     ▼                      ▼                      │
┌──────────────┐   ┌───────────────────┐           │
│ 慢路径 (VLM)  │   │ 快路径 (传统规划)   │           │
│ InternVL-26B │   │ BEVFormer + 3D    │           │
│ → CoT 推理   │   │ 检测 → 预测 → 规划 │           │
│ → 语义决策   │   │ → 精确轨迹         │           │
│ 延迟:300-800ms│   │ 延迟: 80-150ms     │           │
└──────┬───────┘   └───────┬───────────┘           │
       ▼                   ▼                       │
┌──────────────────────────────────────┐           │
│ 融合模块: A.语义条件注入 / B.轨迹加权  │           │
│          / C.安全兜底(VLM override)   │           │
└──────────────────┬───────────────────┘           │
                   ▼
            最终执行轨迹 (延迟: 80-200ms, 快路径主导)
```

### 快慢路径的互补性

| 维度 | 快路径 (传统规划) | 慢路径 (VLM 推理) | 融合效果 |
|------|-----------------|------------------|---------|
| 延迟 | 80-150ms | 300-800ms | 80-200ms (快路径主导) |
| 常规场景 | ✅ 精确几何规划 | ❌ 计算浪费 | 快路径主导 |
| 长尾场景 | ❌ 规则覆盖不足 | ✅ 常识推理 | 慢路径 override |
| 轨迹精度 | ✅ cm 级 | ❌ 10cm 级 | 快路径精度 |
| 可解释性 | ❌ 黑箱/规则 | ✅ 自然语言 | 慢路径解释 |

> 关键洞察：DriveVLM-Dual 的设计哲学类似于 [Helix 双系统架构](../03-models-zoo/09-helix.md)——快系统保持实时响应，慢系统在后台进行深度推理，两者异步协作。

## 🔬 定量实验结果

### nuScenes 规划指标对比

| 方法 | L2 1s(m) | L2 2s(m) | L2 3s(m) | 碰撞率 1s(%) | 碰撞率 3s(%) |
|------|---------|---------|---------|-------------|-------------|
| ST-P3 | 1.33 | 2.11 | 2.90 | 0.23 | 1.27 |
| UniAD | 0.48 | 0.96 | 1.65 | 0.05 | 0.71 |
| VAD-Base | 0.41 | 0.70 | 1.05 | 0.07 | 0.41 |
| DriveVLM (VLM only) | 0.57 | 0.88 | 1.28 | 0.04 | 0.48 |
| DriveVLM-Dual | **0.36** | **0.62** | **0.94** | **0.02** | **0.28** |

**分析**：DriveVLM-Dual 在所有指标上取得最优——VLM 语义理解 + 传统规划器几何精度互补。3s 碰撞率从 UniAD 的 0.71% 降至 0.28%，证明 VLM 风险推理有效。

### 长尾场景专项评测（200+ 长尾场景评测集）

| 场景类型 | 样本数 | DriveVLM 正确率 | VAD 正确率 | 差距 |
|---------|-------|----------------|-----------|------|
| 施工区域 | 45 | 82.2% | 53.3% | +28.9% |
| 异常障碍物 | 38 | 76.3% | 42.1% | +34.2% |
| 特殊交通指示 | 52 | 78.8% | 48.1% | +30.7% |
| 恶劣天气 | 35 | 71.4% | 60.0% | +11.4% |
| 复杂交互 | 42 | 73.8% | 54.8% | +19.0% |

### 延迟分析

| 模型配置 | 硬件 | 视觉编码(ms) | VLM 生成(ms) | 后处理(ms) | 总延迟(ms) |
|---------|------|------------|------------|-----------|-----------|
| DriveVLM-26B | A100 80G | 85 | 650 | 12 | 747 |
| DriveVLM-7B | A100 80G | 45 | 210 | 8 | 263 |
| DriveVLM-Dual | A100 80G | 45 (共享) | 210 (异步) | 8 | ~120 (快路径主导) |
| DriveVLM-7B | H100 80G | 32 | 155 | 6 | 193 |

## 🔬 训练数据与训练流程

### 训练数据构成

| 数据来源 | 规模 | 用途 |
|---------|------|------|
| nuScenes 标注 | 28K 关键帧 + 34K 场景描述 | CoT 三阶段标注 |
| 人工长尾场景标注 | 5K+ 场景 | 长尾场景覆盖 |
| GPT-4V 辅助标注 | 20K+ 场景描述 | 扩充训练数据 |
| 规划轨迹 | 基于 nuScenes GT | 监督轨迹输出 |

### 训练流程

- **Phase 1**：加载 InternVL-Chat 预训练权重，冻结视觉编码器微调 LLM。数据: 场景描述+分析 QA 对。3 epochs, LR=2e-5, batch=32。
- **Phase 2**：解冻全部参数端到端联合训练。数据: 完整 CoT + 轨迹。10 epochs, LR=1e-5, batch=16。Loss = 0.2×L_desc + 0.3×L_analysis + 0.3×L_plan + 0.2×L_traj。

## 意义与局限

### 核心贡献

1. **证明了 CoT 推理在驾驶中的可行性**——VLM 的语言推理确实改善了长尾场景决策
2. **提出了实用的 Dual 架构**——快慢路径融合平衡了推理深度和实时性
3. **建立了长尾场景评测范式**——推动社区关注常规 benchmark 之外的场景

### 局限性

| 局限 | 详情 |
|------|------|
| 真车闭环验证缺失 | 仅在 nuScenes 离线评测，未报告闭环或实车结果 |
| CoT 标注成本高 | 三阶段标注依赖专家+GPT-4V，扩展到大规模数据困难 |
| 26B 模型部署困难 | 车载嵌入式平台（Orin/Thor）无法直接部署 26B 模型 |
| 幻觉风险 | VLM 可能生成与实际场景不符的描述，对安全关键系统有风险 |
| 固定推理模板 | 三阶段 CoT 结构固定，缺乏根据场景复杂度动态调整推理深度的能力 |

> 关于幻觉和安全挑战的深入讨论，参见 [场景理解+决策一体化](./04-scene-understanding.md) 和 [驾驶 VLA 局限与展望](./05-limitations.md)。

## 小结

| 概念 | 要点 |
|------|------|
| 核心思路 | 三阶段 CoT（描述→分析→规划）使驾驶决策可推理可解释 |
| VLM 主干 | InternVL-Chat (InternViT-6B + InternLM2-20B) |
| DriveVLM-Dual | 慢路径(VLM 推理) + 快路径(传统规划) 异步融合 |
| 关键结果 | nuScenes L2 3s: 0.94m (Dual) / 1.28m (VLM only)，长尾场景 +24.8% |
| 延迟 | 26B: 747ms / 7B: 263ms / Dual: ~120ms (快路径主导) |
| 主要局限 | 无真车验证、标注成本高、大模型部署难、幻觉风险 |
