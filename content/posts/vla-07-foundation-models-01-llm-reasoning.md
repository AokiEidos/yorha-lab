# LLM 作为推理引擎

> ⚙️ 进阶 | 前置知识：[从 VLM 到 VLA](../02-architecture/01-vlm-to-vla.md)，[SayCan](../03-models-zoo/01-saycan.md)

## LLM 在 VLA/具身智能中的三大角色

**大语言模型（Large Language Model, LLM）** 为机器人系统提供了三种此前不可能的能力：**任务理解与分解、常识推理、思维链决策**。这些能力不是从机器人数据中学到的，而是从互联网规模的文本训练中"继承"的——这是 VLA 泛化能力的关键来源之一。

## 角色一：任务理解与分解

### 从模糊指令到可执行子任务

传统机器人只能理解预定义的、精确的指令（如 `move_to(x=0.3, y=0.2, z=0.1)`）。LLM 可以理解人类自然的、模糊的、隐含信息的指令，并将其分解为具体步骤：

```
用户指令: "帮我准备明天的午餐便当"
                │
    [LLM 推理分解]
                │
                ▼
子任务序列:
  1. 打开冰箱取出食材（推理出需要的食材类型）
  2. 清洗蔬菜（推理出午餐便当通常含蔬菜）
  3. 加热米饭（推理出便当需要主食）
  4. 将菜品分装到便当盒中（推理出容器）
  5. 盖上盖子，放入保温袋（推理出保鲜需求）
```

注意 LLM 在这里做了大量**隐含推理**——用户没有说要什么食材、用什么容器、怎么保鲜，LLM 从常识中补全了这些信息。

### 分解的层级

LLM 的分解通常形成层级结构：

```
高层目标:  "整理厨房"
              │
    ┌────────┼────────┐
中层子任务:  清理台面    洗碗        擦灶台
              │         │           │
    ┌────┬───┘   ┌────┬┘     ┌────┬┘
底层动作: 拿起碗  放入水槽  拿起碗 打开水龙头 拿海绵 擦拭
```

SayCan、Inner Monologue 等系统利用这种层级分解来桥接 LLM 的高层理解和 VLA 的底层执行。

### 代表性系统对比

| 系统 | LLM | 分解方式 | 执行层 | 反馈机制 |
|------|-----|---------|--------|---------|
| **SayCan** | PaLM | 候选动作打分 | 预训练 RL 技能 | 无 |
| **Inner Monologue** | PaLM | 自由文本分解 | 预训练 RL 技能 | 成功/失败反馈→重新规划 |
| **Code-as-Policies** | Codex/GPT-4 | 生成 Python 代码 | 代码执行器 | 异常处理 |
| **ProgPrompt** | GPT-3 | 结构化程序 | 仿真执行器 | 断言检查 |
| **TidyBot** | GPT-4 | 物品分类+放置规则 | 移动操作策略 | 用户纠正 |

### Code-as-Policies：LLM 写代码控制机器人

**Code-as-Policies（CaP）**（Liang et al., 2022）让 LLM 直接生成 Python 代码来控制机器人——比自然语言分解更精确：

```python
# LLM 生成的机器人控制代码示例
# 用户指令: "把所有红色的方块排成一行"

red_blocks = detect_objects(color="red", shape="block")
red_blocks_sorted = sorted(red_blocks, key=lambda b: b.position.x)
target_y = 0.3  # 排成一行的 y 坐标
for i, block in enumerate(red_blocks_sorted):
    target_x = 0.2 + i * 0.08  # 间隔 8cm
    pick_and_place(block, Position(x=target_x, y=target_y, z=0.02))
```

优势：代码是精确的、可执行的、可调试的。
劣势：需要预定义 API（`detect_objects`、`pick_and_place`），灵活性受限。

## 角色二：常识推理

### 互联网知识→物理世界决策

LLM 从互联网训练数据中积累的**常识知识（Commonsense Knowledge）** 可以直接辅助机器人决策——这是传统方法完全不具备的能力。

| 常识类别 | 具体示例 | 对机器人的价值 | 传统方法能否获得 |
|---------|---------|-------------|--------------|
| **物理属性** | "玻璃杯易碎，塑料杯不怕摔" | 调整抓取力度和放置小心程度 | ❌ 需要专门的材质数据库 |
| **空间常识** | "盘子通常放在碗柜里，不在冰箱里" | 辅助物品搜索/导航 | ❌ 需要逐个场景建图 |
| **功能常识** | "海绵用来擦东西，不是用来切东西" | 工具选择 | ❌ 需要工具知识图谱 |
| **社会常识** | "递剪刀要把柄朝向对方" | 安全的人机交互 | ❌ 需要专门编程 |
| **因果推理** | "杯子倒了→水会洒出来→需要擦拭" | 预见行动后果 | ❌ 需要物理仿真 |
| **时间常识** | "先烧水，再泡茶（不能反过来）" | 任务排序 | ❌ 需要手工定义先后依赖 |

### RT-2 中的常识推理验证

RT-2 论文中最引人注目的实验之一：

```
指令: "Pick up the object that is not healthy to eat"
场景: [苹果, 薯片, 饼干]
      
模型推理过程（隐式）:
  - 识别三个物体
  - 从互联网知识中知道：苹果健康，薯片和饼干不太健康
  - 薯片通常被认为最不健康
  - 输出：抓取薯片的动作

训练数据中从未有过"判断食物健康程度→抓取"的机器人演示！
```

这说明 VLM 的互联网知识确实可以通过推理链路影响机器人动作——即**涌现（Emergence）**。

## 角色三：思维链（Chain-of-Thought）决策

### CoT 在机器人中的形式

**思维链（Chain-of-Thought, CoT）** 推理让模型在输出动作前先生成中间推理步骤。在机器人场景中，CoT 可以有多种形式：

#### 形式 1：自然语言推理

```
观测: [图像：桌上有一个倒扣的杯子和一瓶水]
指令: "倒一杯水"

CoT 推理:
  "分析：
   1. 目标是给杯子倒水
   2. 但杯子是倒扣的——如果直接倒水会洒出来
   3. 所以第一步应该先把杯子翻过来
   4. 翻杯子时：从侧面抓住杯底，旋转 180° 使杯口朝上
   5. 然后拿起水瓶，对准杯口
   6. 倾斜水瓶约 45°，缓慢倒水
   7. 观察水位到八分满时停止
   
  执行计划：翻转杯子 → 拿水瓶 → 倒水 → 观察水位 → 停止"

动作输出: [抓取杯底的运动轨迹...]
```

#### 形式 2：结构化状态描述

```json
{
  "scene_description": "木桌上有3个物体：红色马克杯(位于左前方)、蓝色积木(中央)、黄色香蕉(右侧)",
  "goal_analysis": "指令要求把水果放到杯子旁边。香蕉是水果，马克杯是杯子。",
  "spatial_reasoning": "香蕉在右侧(x=0.4), 马克杯在左前方(x=0.1, y=0.3)。需要把香蕉移到x≈0.15, y≈0.3",
  "plan": ["接近香蕉", "抓取香蕉", "移动到马克杯旁", "放置"],
  "action": [0.12, -0.05, 0.02, 0, 0, 0, 1]
}
```

#### 形式 3：Inner Monologue（闭环推理）

**Inner Monologue（内部独白）**（Huang et al., 2022）的关键创新：在执行过程中持续自我对话，根据执行结果调整计划：

```
[规划] "第一步：拿起红色方块"
[执行] VLA 执行抓取动作
[感知反馈] 视觉检测：手中无物体 → "抓取失败"
[重新推理] "抓取失败了，可能是位置偏了。让我重新定位——
            红色方块看起来比预想的更靠右。调整目标位置 +2cm。"
[再次执行] VLA 重新抓取
[感知反馈] 本体感觉：夹爪力度增加 → "成功抓取"
[继续] "好的，方块拿到了。下一步：移动到蓝色碗上方..."
```

这种闭环推理使 LLM 具备了**自我纠错**能力——不再是"规划一次就执行到底"，而是持续监控和调整。

## 推理延迟：LLM 作为推理引擎的核心瓶颈

### 延迟分析

| 模型 | 参数量 | 硬件 | 首 Token 延迟 | 生成速度 | 100 Token 总延迟 | 等效控制频率 |
|------|--------|------|-------------|---------|---------------|------------|
| Llama 2 7B | 7B | A100 | ~50ms | ~40 tok/s | ~2.5s | <1 Hz |
| Gemma 2B | 2.5B | A100 | ~20ms | ~80 tok/s | ~1.3s | <1 Hz |
| Llama 2 7B | 7B | Jetson Orin | ~200ms | ~8 tok/s | ~12.5s | <0.1 Hz |
| Gemma 2B | 2.5B | Jetson Orin | ~80ms | ~20 tok/s | ~5s | ~0.2 Hz |

对比机器人控制需求：桌面操作需要 5-10 Hz，行走平衡需要 50-200 Hz。**LLM 推理速度远远不够直接做实时控制。**

### 缓解策略详解

#### 策略 1：异步推理（Asynchronous Reasoning）

```
时间线:
  LLM:   [───推理───][───推理───][───推理───]
                                              低频 (~1 Hz)
  底层策略: [动作][动作][动作][动作][动作][动作][动作]
                                              高频 (~10-50 Hz)
  
  LLM 的规划结果作为"高层目标"异步传递给底层策略
```

#### 策略 2：推测解码（Speculative Decoding）

用小模型先快速生成草稿，大模型验证——在保持大模型质量的同时获得接近小模型的速度。

#### 策略 3：动作 Token 缓存

```python
# 伪代码：缓存相似场景的推理结果
class ActionCache:
    def get_action(self, observation, instruction):
        # 查找相似场景的缓存
        key = encode_scene(observation, instruction)
        cached = self.cache.nearest_neighbor(key, threshold=0.95)
        if cached:
            return cached.action  # 命中缓存，跳过 LLM 推理
        else:
            action = self.llm.reason(observation, instruction)  # 未命中，完整推理
            self.cache.store(key, action)
            return action
```

#### 策略 4：双系统架构（Dual-System）

如 Helix 和 GR00T N1 采用的方案：

```
System 2 (LLM, ~1 Hz): "把杯子放到桌子左边" → 高层目标
                              │ 异步下发
                              ▼
System 1 (快速网络, ~50 Hz): 接收目标 → 实时生成关节命令
                              持续执行直到目标完成或新目标到达
```

这是当前最实用的方案——将 LLM 的推理价值（常识、分解、推理）与实时控制的需求解耦。

## LLM 推理的局限性

| 局限 | 描述 | 影响 |
|------|------|------|
| **幻觉（Hallucination）** | LLM 可能"编造"不存在的物体或不合理的物理关系 | 机器人执行不存在的动作 |
| **空间推理弱** | LLM 对精确空间关系（"左边 3cm"）的推理不可靠 | 精细操作失败 |
| **物理推理有限** | LLM 对力学、流体等物理过程的推理不精确 | 力控制/流体操作失败 |
| **不可靠性** | 同一输入可能产生不同输出 | 行为不一致 |
| **上下文长度限制** | 长时域任务可能超出上下文窗口 | 长任务遗忘前期信息 |

## 小结

| 概念 | 要点 |
|------|------|
| 任务分解 | LLM 将模糊指令→具体子任务序列（多层级） |
| 常识推理 | 物理/空间/功能/社会常识辅助决策 |
| CoT 推理 | 自然语言/结构化/Inner Monologue 三种形式 |
| 推理延迟 | 核心瓶颈：7B 模型 ~2.5s/次（A100） |
| 缓解方案 | 异步推理 / 缓存 / 双系统架构 |
| 局限 | 幻觉、空间推理弱、物理推理有限 |
