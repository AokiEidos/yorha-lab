# 采样加速概述

> ⚙️ 进阶 | 前置知识：[采样算法](../01-fundamentals/04-sampling-algorithms.md)，[DDIM 详解](../02-models-zoo/02-ddim.md)

## 为什么加速是扩散模型的核心瓶颈 🔰

**扩散模型（Diffusion Model）** 的生成质量已经超越了 GAN，但其推理效率仍然是生产部署的最大障碍。要理解这个问题的严重性，我们需要从基础数学出发。

### 推理开销的数学分析

标准 DDPM 的逆向过程定义为：

$$x_{t-1} = \frac{1}{\sqrt{\alpha_t}}\left(x_t - \frac{1-\alpha_t}{\sqrt{1-\bar{\alpha}_t}}\epsilon_\theta(x_t, t)\right) + \sigma_t z$$

每一步都需要一次完整的 U-Net **前向传播（Forward Pass）**，即一次**网络函数评估（Network Function Evaluation, NFE）**。以 Stable Diffusion 1.5 为例，U-Net 包含约 860M 参数，其推理开销如下：

| 配置 | 单步耗时 | 1000 步总耗时 | 50 步 DDIM + CFG | 等效 NFE |
|------|---------|-------------|----------------|---------|
| A100 80GB | ~30ms | ~30s | ~3.0s | 100 (CFG 双倍) |
| RTX 4090 | ~45ms | ~45s | ~4.5s | 100 |
| RTX 3060 | ~120ms | ~120s | ~12s | 100 |
| Jetson Orin | ~250ms | ~250s | ~25s | 100 |

> 注意：使用 **CFG（Classifier-Free Guidance）** 时，每步需要同时计算条件预测和无条件预测，NFE 翻倍。

### 与其他生成模型的对比

| 模型 | 单张生成耗时 (A100) | 质量 (FID↓) | 可控性 |
|------|-------------------|-----------|--------|
| GAN (StyleGAN2) | ~50ms | 2.92 | 有限 |
| VAE (VQGAN) | ~20ms | 5.20 | 有限 |
| DDPM 1000步 | ~30s | 2.10 | 高 |
| DDIM 50步 + CFG | ~3s | 2.50 | 高 |
| 理想目标 | <100ms | <3.0 | 高 |

扩散模型与 GAN 之间存在约 **60-600 倍**的推理速度差距。这意味着：
- **实时应用**（交互式编辑、视频生成）基本不可行
- **大规模服务**成本极高：百万级日请求量下，GPU 算力成本可能达到数万美元/天
- **端侧部署**（手机、嵌入式设备）几乎不可能

这个巨大的差距驱动了过去两年密集的加速研究。

## 三大加速方向与技术分类 ⚙️

扩散模型的加速可以从三个正交的维度入手，每个维度独立降低推理延迟：

```
                    扩散模型加速技术分类
                          │
          ┌───────────────┼───────────────┐
          │               │               │
    ① 减少步数        ② 加速单步        ③ 减少计算量
     (NFE↓)          (ms/step↓)        (FLOPs↓)
          │               │               │
    ┌─────┴─────┐   ┌─────┴─────┐   ┌─────┴─────┐
    │           │   │           │   │           │
  ODE求解器  蒸馏  TensorRT  Flash   量化    Token
  DPM-Solver LCM  编译加速  Attn   INT8/4  Merging
  UniPC     ADD   ONNX RT  xFormers W4A8   DeepCache
  Heun      DMD2  CUDA Graph        剪枝   特征缓存
```

### 方向一：减少采样步数（NFE 优化）

**核心思想**：用更聪明的策略减少去噪步数，同时维持生成质量。

- **高阶 ODE 求解器**：将 DDIM 的一阶 Euler 方法升级为 2-3 阶方法（DPM-Solver, Heun），用更少的步数达到相同的数值精度。典型效果：50 步 → 10-20 步。
- **蒸馏方法**：训练学生模型，让 1 步学生 ≈ 多步教师。典型效果：50 步 → 1-4 步。包括渐进式蒸馏、一致性蒸馏（LCM）、对抗蒸馏（ADD）等。

### 方向二：加速单步推理

**核心思想**：在不改变模型行为的前提下，优化每一步网络前向传播的执行速度。

- **编译优化**：TensorRT / ONNX Runtime / torch.compile 将 PyTorch 模型编译为优化后的推理引擎（算子融合、内存规划、kernel 自动调优）。
- **高效算子**：Flash Attention、xFormers 等针对注意力计算的高效实现。
- **CUDA Graph**：消除 kernel launch 开销，特别适合小 batch 推理。

### 方向三：减少计算量

**核心思想**：通过降低精度或减少参与计算的元素数量来降低 FLOPs。

- **量化**：FP16 → INT8 → INT4，减少内存带宽需求和计算量。
- **特征缓存**：DeepCache 等方法利用相邻步特征的相似性，跳过部分计算。
- **Token 压缩**：ToMe（Token Merging）合并相似 token，减少序列长度。
- **模型剪枝/架构搜索**：去除冗余参数或设计更紧凑的架构。

## 加速效果综合对比表 🔬

以下表格展示了各方法在 Stable Diffusion 1.5（512x512 分辨率）上的典型效果：

| 方法 | 类别 | 步数 | 每步耗时 (A100) | 总耗时 | FID 变化 | 额外训练 |
|------|------|------|----------------|--------|---------|---------|
| DDPM 基线 | — | 1000 | ~30ms | ~30s | 基准 | — |
| DDIM + CFG 7.5 | 求解器 | 50 (100 NFE) | ~30ms | ~3.0s | +0.3 | 否 |
| DPM-Solver++ 2M | 求解器 | 20 (40 NFE) | ~30ms | ~1.2s | +0.1 | 否 |
| DPM-Solver++ 2M + TRT | 求解器+编译 | 20 (40 NFE) | ~12ms | ~0.48s | +0.1 | 否 |
| LCM 4步 | 蒸馏 | 4 (4 NFE) | ~30ms | ~0.12s | +1.5 | 是 |
| LCM 4步 + TRT + FA | 蒸馏+编译 | 4 (4 NFE) | ~15ms | ~0.06s | +1.5 | 是 |
| SDXL-Turbo 1步 | 蒸馏 | 1 (1 NFE) | ~65ms | ~0.065s | +2.0 | 是 |
| LCM + INT8 + TRT | 全栈 | 4 (4 NFE) | ~10ms | ~0.04s | +2.0 | 是 |

> 注：FID 变化是相对于 50 步 DDIM 基线的增量，越小越好。实际数值因模型版本和评测设置有所不同。

## 方法可组合性分析 ⚙️

三个方向的加速方法大部分可以正交组合，但也存在一些冲突：

| 组合 | 兼容性 | 说明 |
|------|--------|------|
| ODE 求解器 + TensorRT | 完全兼容 | 编译不影响求解器逻辑 |
| ODE 求解器 + Flash Attention | 完全兼容 | 算子级优化，透明替换 |
| ODE 求解器 + 量化 | 完全兼容 | 但低步数下量化误差累积更明显 |
| ODE 求解器 + 蒸馏 | **冲突** | 蒸馏模型有专用步数，不适用高阶求解器 |
| 蒸馏 + TensorRT | 完全兼容 | 推荐组合 |
| 蒸馏 + Flash Attention | 完全兼容 | 推荐组合 |
| 蒸馏 + 量化 | 需要适配 | 蒸馏模型可能需要重新校准量化参数 |
| 蒸馏 + DeepCache | **部分冲突** | 步数已经很少时缓存收益有限 |
| TensorRT + Flash Attention | 需要适配 | TRT 有自己的注意力优化，需确认版本兼容 |
| 量化 + DeepCache | 完全兼容 | 可以同时使用 |
| Token Merging + Flash Attention | 完全兼容 | 序列变短后 FA 更快 |

**最佳实践组合**：

```
高性能生产部署:  蒸馏(LCM 4步) + TensorRT + FP16 + Flash Attention
                → ~60ms/张 (A100), ~100ms/张 (RTX 4090)

平衡质量与速度:  DPM-Solver++ 20步 + TensorRT + FP16 + Flash Attention
                → ~480ms/张 (A100), ~800ms/张 (RTX 4090)

极致低延迟:     SDXL-Turbo 1步 + TensorRT + INT8
                → ~30ms/张 (A100), ~50ms/张 (RTX 4090)
```

## 生产部署成本分析 🔬

对于大规模图像生成服务，加速方案的选择直接影响运营成本。以下以日均 100 万张图像的生成服务为例：

### 成本模型

$$\text{GPU 数量} = \frac{\text{日请求量} \times \text{单张延迟}}{\text{每日可用秒数} \times \text{GPU 利用率}}$$

假设 GPU 利用率为 70%，每日可用秒数 = 86400：

| 方案 | 单张延迟 (A100) | 所需 A100 数量 | 月成本 (按 $2/GPU-hr) | 年成本 |
|------|----------------|--------------|---------------------|--------|
| DDIM 50步 + FP16 | 3.0s | ~50 | ~$72,000 | ~$864,000 |
| DPM-Solver++ 20步 + TRT | 0.48s | ~8 | ~$11,520 | ~$138,240 |
| LCM 4步 + TRT + FA | 0.06s | ~1 | ~$1,440 | ~$17,280 |
| SDXL-Turbo 1步 + TRT | 0.035s | ~1 | ~$1,440 | ~$17,280 |

从 DDIM 基线到 LCM+TRT 全栈优化，**成本降低了约 50 倍**。这不仅是工程优化，更是决定扩散模型能否商业化的关键因素。

### 延迟 vs 质量的权衡曲线

```
FID (↓更好)
 │
 │  * DDPM 1000步 (最优质量)
3│  * DDIM 50步
 │    * DPM-Solver++ 20步
 │
5│                          * LCM 4步
 │                          
 │                            * SDXL-Turbo 1步
7│
 │
 └──────────────────────────────────────── 延迟 (log)
    30s    3s    1s   0.1s  0.05s  0.03s
```

没有"最佳"方案——只有适合特定场景的最佳方案。延迟要求越严格，需要接受的质量损失越大。

### 边缘设备部署的成本考量

对于移动端和嵌入式场景，核心约束不是 GPU 租赁成本，而是**功耗**和**内存**：

| 平台 | 内存预算 | 功耗预算 | 可行方案 | 预期延迟 |
|------|---------|---------|---------|---------|
| 旗舰手机 (Snapdragon 8 Gen3) | ~4 GB | ~5W | SD 1.5 INT4 + 4步 LCM | ~3s |
| 中端手机 (Snapdragon 7 Gen1) | ~2 GB | ~3W | Tiny SD INT4 + 4步 | ~8s |
| Jetson Orin Nano | ~4 GB (共享) | ~15W | SD 1.5 INT8 + TRT | ~5s |
| Raspberry Pi 5 | ~4 GB | ~5W | 不可行 (无 GPU) | — |

移动端部署通常需要**模型压缩 + 架构简化 + 量化**的三重优化。

## 加速技术发展时间线 ⚙️

```
2020.06  DDPM (Ho et al.)          — 1000 步基线
2020.10  DDIM (Song et al.)        — 50 步，首个加速方法
2022.03  Progressive Distillation  — 4 步蒸馏
2022.06  DPM-Solver (Lu et al.)    — 10-20 步高阶求解器
2022.10  Karras et al. Sampler     — 优化噪声调度
2022.11  DPM-Solver++ (Lu et al.)  — CFG 友好的求解器
2023.03  Consistency Models (Song) — 1-2 步一致性训练
2023.05  Flash Attention 2 (Dao)   — 注意力 3-4× 加速
2023.08  LCM (Luo et al.)         — 4 步潜空间一致性
2023.10  LCM-LoRA                  — 轻量级一致性适配
2023.11  SDXL-Turbo / ADD          — 1 步对抗蒸馏
2024.01  DeepCache (Ma et al.)     — 特征缓存 2× 加速
2024.03  DMD2 (Yin et al.)        — 分布匹配蒸馏
2024.06  SD 3.0 + DiT 架构加速     — 新架构的优化挑战
```

趋势：从数值方法优化 → 知识蒸馏 → 工程优化 → 三者融合。每 6-12 个月，端到端延迟降低一个数量级。

## SDXL 与更大模型的加速挑战 ⚙️

SDXL（2.6B 参数）和 SD 3.0/Flux（MM-DiT 架构）带来了新的加速挑战：

| 挑战 | 说明 | 应对策略 |
|------|------|---------|
| 参数量增大 | SDXL 是 SD1.5 的 3 倍 | 量化的收益更大 |
| 注意力层更多 | DiT 架构全是注意力 | Flash Attention 更关键 |
| 更高分辨率 | 1024x1024 默认 | VAE Tiling、Token Merging |
| Refiner 流程 | SDXL 的两阶段 | 可考虑跳过或蒸馏 Refiner |
| CFG scale 更高 | 通常 7-12 | 引导蒸馏的收益更大 |
