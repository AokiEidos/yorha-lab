# 端到端驾驶中的 VLA 范式

> ⚙️ 进阶 | 前置知识：[从 VLM 到 VLA](../02-architecture/01-vlm-to-vla.md)、[动作表征](../02-architecture/02-action-representation.md)

## 为什么自动驾驶需要 VLA

**视觉-语言-动作模型（Vision-Language-Action Model, VLA）** 将视觉感知、语言推理和动作生成统一到单一网络中。在自动驾驶场景下，这一范式解决了两个根本性问题：(1) 模块化流水线中的信息瓶颈；(2) 对长尾场景（**长尾场景（Long-tail Scenario）**，指训练数据中极少出现但现实中偶发的驾驶情境）的处理能力不足。

本文从三种自动驾驶技术路线的对比出发，深入分析 VLA 范式的独特定位、系统架构、传感器处理方式以及当前基准评测体系。

## 🔰 三种驾驶架构对比

### 传统模块化流水线

以 Apollo（百度）和 Autoware（开源）为代表的模块化架构，将自动驾驶拆分为感知、预测、规划、控制四大模块：

```
┌────────────────────────────────────────────────────────────────────────┐
│                    传统模块化架构 (Apollo / Autoware)                    │
├──────────┬──────────┬──────────┬──────────┬──────────┬────────────────┤
│ 传感器    │ 感知模块  │ 预测模块  │ 规划模块  │ 控制模块  │  执行器        │
│ Camera   │ 3D 检测  │ 轨迹预测  │ 路径搜索  │ PID/MPC  │  转向/油门     │
│ LiDAR    │ 语义分割  │ 意图分类  │ 速度规划  │ 横纵向   │  刹车         │
│ Radar    │ 车道检测  │ 交互建模  │ 决策树    │ 解耦控制  │              │
│ IMU/GPS  │ 红绿灯   │          │          │          │              │
├──────────┼──────────┼──────────┼──────────┼──────────┼────────────────┤
│延迟贡献   │ 50-80ms  │ 20-40ms  │ 30-60ms  │ 5-10ms   │ 10-20ms      │
│          │          │          │          │          │ 总计:115-210ms │
└──────────┴──────────┴──────────┴──────────┴──────────┴────────────────┘
```

**优势**：每个模块可独立开发、测试和替换；几何规划器可提供数学级安全保证。

**劣势**：模块间以结构化数据（检测框、轨迹点）交接，原始感知信息大量丢失；误差逐级累积——nuScenes 上 3D 检测 mAP 约 65%，预测 minADE 约 1.5m，规划 L2 误差约 2.5m（3s），误差链条导致末端规划质量受限。

### 端到端纯视觉方案

以 **UniAD**（Hu et al., CVPR 2023）和 **VAD**（Jiang et al., ICCV 2023）为代表，用单一网络完成从传感器到规划的全过程：

```
┌──────────────────────────────────────────────────┐
│           端到端方案 (UniAD / VAD)                 │
├─────────────────┬────────────────────────────────┤
│ 多相机图像输入    │  BEV 编码器                     │
│ 6 cameras       │  (BEVFormer / LSS)             │
│ 1600×900 each   │         │                      │
│                 │         ▼                      │
│                 │  ┌─────────────────────┐       │
│                 │  │ 联合 Query 解码器     │       │
│                 │  │ Track + Map + Motion │       │
│                 │  │ + Occupancy + Plan   │       │
│                 │  └─────────┬───────────┘       │
│                 │            ▼                    │
│                 │    规划轨迹 (x,y) × T steps     │
├─────────────────┴────────────────────────────────┤
│ 延迟: 约 150-300ms (A100 单次前向)                  │
└──────────────────────────────────────────────────┘
```

**nuScenes 规划基准**（L2 距离，单位 m）：

| 模型 | 1s | 2s | 3s | 平均碰撞率(%) |
|------|-----|-----|-----|-------------|
| ST-P3 | 1.33 | 2.11 | 2.90 | 0.23 |
| UniAD | 0.48 | 0.96 | 1.65 | 0.05 |
| VAD-Base | 0.41 | 0.70 | 1.05 | 0.07 |

**优势**：端到端梯度流通，信息不再在模块间丢失。**劣势**：缺乏语言理解能力——无法处理"前方施工请绕行"的语音指令；无法解释自身决策；对完全陌生的场景（如交警手势指挥）泛化有限。

### ⚙️ VLA 驾驶范式

VLA 在端到端的基础上引入语言通道，赋予系统三项独特能力：

```
┌───────────────────────────────────────────────────────────────┐
│                     VLA 驾驶架构                               │
│                                                               │
│  ┌─────────────┐   ┌──────────────┐   ┌──────────────┐       │
│  │ 多视角图像    │   │ 导航指令      │   │ 实时对话      │       │
│  │ (6 cameras)  │   │"在下个路口    │   │"前面那个人    │       │
│  │              │   │  左转"        │   │ 是在过马路吗" │       │
│  └──────┬───────┘   └──────┬───────┘   └──────┬───────┘       │
│         │                  │                  │               │
│         ▼                  ▼                  ▼               │
│  ┌─────────────────────────────────────────────────┐          │
│  │              VLM 主干 (VLA Core)                 │          │
│  │  ┌──────────┐  ┌──────────────┐  ┌───────────┐ │          │
│  │  │ 视觉编码器│  │ 语言编码器    │  │ 跨模态融合 │ │          │
│  │  │ViT-L/14  │  │ LLaMA-7B    │  │ Q-Former  │ │          │
│  │  └────┬─────┘  └──────┬───────┘  └─────┬─────┘ │          │
│  │       └───────────────┼─────────────────┘       │          │
│  │                       ▼                         │          │
│  │            ┌──────────────────┐                  │          │
│  │            │  CoT 推理链       │                  │          │
│  │            │ 场景描述→分析→规划 │                  │          │
│  │            └────────┬─────────┘                  │          │
│  └─────────────────────┼───────────────────────────┘          │
│                        ▼                                      │
│  ┌──────────────────────────────────────┐                     │
│  │ 动作输出                              │                     │
│  │ 轨迹点 [(x₁,y₁,t₁), ...] 或         │                     │
│  │ 控制命令 (steering, throttle, brake)  │                     │
│  └──────────────────────────────────────┘                     │
└───────────────────────────────────────────────────────────────┘
```

## ⚙️ VLA 的五大独特能力与具体场景

| # | 能力 | 纯视觉端到端 | VLA | 具体驾驶场景 |
|---|------|------------|-----|------------|
| 1 | 语言指令理解 | ❌ | ✅ | 乘客说"我赶时间，走高速"——VLA 理解语义并重新规划路线 |
| 2 | 常识推理 | 有限 | ✅ | 看到学校标志+放学时间——推理出可能有大量儿童横穿，主动降至 20km/h |
| 3 | 异常场景处理 | 差 | ✅ | 前方车辆后备箱掉落货物——识别非标准障碍物并绕行 |
| 4 | 可解释决策 | ❌ | ✅ | 急刹后解释"检测到右侧行人突然进入车道，触发紧急制动" |
| 5 | 多轮交互 | ❌ | ✅ | 乘客问"为什么不超车"，系统回答"前方弯道视距不足，超车不安全" |

### 场景实例：施工区域通行

```
输入图像: [前方 200m 处锥桶、施工围栏、工人手持停车牌]
导航指令: "继续沿当前路线行驶"

VLA CoT 推理:
  Step 1 (场景描述): "前方检测到施工区域——左侧车道被锥桶封闭，
                     右侧有施工围栏。一名工人手持红色停车牌。"
  Step 2 (场景分析): "工人指示停车等待。施工区域约 100m 长。
                     风险等级: 高。关键实体: 工人(最高优先级)。"
  Step 3 (决策):     "执行停车，等待工人放行信号。"

输出动作: throttle=0.0, brake=0.8, steering=0.0
```

## ⚙️ 传感器输入与处理

### 多相机 BEV 融合 vs 单图像输入

当前驾驶 VLA 在传感器输入上存在两种路线：

| 路线 | 代表方法 | 输入 | BEV 构建 | 优势 | 劣势 |
|------|---------|------|---------|------|------|
| 多相机 BEV | DriveVLM, UniAD | 6 路环视 + 时序帧 | BEVFormer / LSS | 完整 360 度感知 | 计算量大，VLM 需适配 |
| 单/少量图像 | GPT-Driver, DriveGPT4 | 前视 1-3 路 | 不构建 BEV | VLM 原生支持 | 感知视野有限 |
| 图像+LiDAR | LMDrive | 多路图像+LiDAR | 点云 BEV + 图像 | 深度信息精确 | 传感器成本高 |

**关键设计决策**：多相机 BEV 路线在空间感知上更完整，但将 BEV 特征送入 VLM 需要额外的适配层（如 Q-Former 或线性投影），增加了训练复杂度。单图像路线对 VLM 更友好，但在需要 360 度态势感知的驾驶场景中表现受限。

### 延迟分析：模块化 vs 端到端 vs VLA

```
模块化 (Apollo):
  感知(70ms) → 预测(30ms) → 规划(45ms) → 控制(8ms) = ~153ms
  ✅ 各模块可并行优化   ❌ 串行延迟累加

端到端 (UniAD):
  前向传播(200ms) → 后处理(20ms) = ~220ms
  ✅ 单次推理   ❌ 模型大，难以进一步压缩

VLA (DriveVLM, 7B):
  视觉编码(80ms) → VLM 推理+生成(300-800ms) → 解析(10ms) = ~390-890ms
  ✅ 深度推理能力   ❌ 自回归生成是延迟瓶颈

VLA (DriveVLM, 蒸馏 1B):
  视觉编码(40ms) → VLM 推理(80-150ms) → 解析(5ms) = ~125-195ms
  ✅ 接近实时   ❌ 推理深度受限
```

## 🔬 安全架构：VLA 与 RSS/安全监控的协同

VLA 模型本质是概率性的——它基于训练数据的统计规律做决策，无法提供形式化的安全保证。在安全关键的自动驾驶中，需要与**责任敏感安全模型（Responsibility-Sensitive Safety, RSS）** 等确定性安全机制协同：

```
┌─────────────────────────────────────────────────┐
│                安全分层架构                        │
│                                                  │
│  Layer 3: VLA 推理层 (慢，深度推理)                │
│  ┌────────────────────────────────────────┐      │
│  │ DriveVLM: CoT → 决策 + 轨迹           │      │
│  │ 延迟: 300-800ms  更新频率: 1-3 Hz      │      │
│  └──────────────────┬─────────────────────┘      │
│                     ▼                            │
│  Layer 2: 轨迹安全校验 (RSS / 可达集分析)           │
│  ┌────────────────────────────────────────┐      │
│  │ 输入: VLA 规划轨迹                      │      │
│  │ 校验: 是否满足安全距离/加速度约束         │      │
│  │ 输出: 通过 → 执行 / 拒绝 → 回退安全轨迹  │      │
│  │ 延迟: <5ms                              │      │
│  └──────────────────┬─────────────────────┘      │
│                     ▼                            │
│  Layer 1: 紧急安全层 (硬件级)                      │
│  ┌────────────────────────────────────────┐      │
│  │ AEB 自动紧急制动 / TTC 碰撞时间检测      │      │
│  │ 延迟: <20ms  独立于上层软件              │      │
│  └────────────────────────────────────────┘      │
└─────────────────────────────────────────────────┘
```

这种分层架构确保：即使 VLA 产生错误决策（幻觉/延迟），底层安全机制仍能保障车辆不会进入危险状态。

## 🔬 当前基准评测体系

### 主流 Benchmark 总览

| Benchmark | 规模 | 场景 | 评测维度 | 适用范围 |
|-----------|------|------|---------|---------|
| **nuScenes** | 1000 场景, 140万 3D 框 | 波士顿/新加坡 | 检测/跟踪/预测/规划 | 离线开环评测 |
| **Waymo Open** | 1150 场景, 1200万 3D 框 | 美国多城市 | 检测/跟踪/预测 | 离线开环评测 |
| **CARLA** | 无限(仿真) | 可定制 | 闭环驾驶成功率/路线完成率 | 仿真闭环评测 |
| **nuPlan** | 1500h 驾驶日志 | 波士顿/新加坡/匹兹堡/拉斯维加斯 | 闭环规划 | 规划专项评测 |
| **DriveLM** | 基于 nuScenes 扩展 | 同 nuScenes | QA + 规划 | VLA 专项评测 |
| **CARLA Leaderboard 2.0** | 标准化路线 | 仿真 | 路线完成率/违规率 | 闭环综合评测 |

### VLA 评测的特殊挑战

传统自动驾驶 benchmark 面向感知/规划的数值指标（mAP, L2 距离），但 VLA 的核心优势——语言推理和可解释性——缺乏标准化评测方法。当前社区正在尝试：

- **DriveLM**：为 nuScenes 场景标注 QA 对，评测 VLA 的感知问答和推理能力
- **Reason2Drive**：专注于驾驶推理能力的评测，包含因果关系问答
- **DRAMA**：评测风险感知和语言解释质量

这些新 benchmark 推动 VLA 从"能不能开好"向"能不能说清楚为什么这样开"拓展评测维度。

## 当前成熟度评估

VLA 在自动驾驶中的应用**显著早于其在机器人操作中的成熟度**。核心原因：

1. **延迟要求极端**：120km/h 时每 100ms 车辆移动 3.3m，VLM 的自回归生成难以满足
2. **安全标准严苛**：ASIL-D 级功能安全要求失效率 <10^-8/h，概率模型难以证明
3. **闭环验证不足**：多数工作仅在 nuScenes 离线评测或 CARLA 仿真中验证
4. **数据规模有限**：驾驶 VLA 训练数据（万级场景）远小于通用 VLM 预训练数据（十亿级图文对）

> 参考：机器人操作领域的 VLA（如 [RT-2](../03-models-zoo/03-rt2.md)、[OpenVLA](../03-models-zoo/05-openvla.md)）已在真实机器人上实现 >80% 的任务成功率，而驾驶 VLA 尚未有公开的真车闭环成功率报告。

## 小结

| 概念 | 要点 |
|------|------|
| 三种架构定位 | 模块化(可靠但信息损失) → 端到端(信息保留但不可解释) → VLA(推理+解释) |
| VLA 五大独特能力 | 语言理解、常识推理、异常场景处理、可解释决策、多轮交互 |
| 延迟现状 | 7B VLA ~400-900ms，蒸馏 1B ~125-195ms，距 <100ms 实时要求仍有差距 |
| 安全架构 | VLA 推理层 + RSS 安全校验 + 硬件 AEB 三层防护 |
| 基准评测 | nuScenes/CARLA 为主，DriveLM 等 VLA 专项 benchmark 正在发展 |
| 成熟度 | 早期研究阶段，最可能作为辅助推理模块而非独立驾驶系统 |
