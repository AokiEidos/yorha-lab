# 具身智能 Agent 框架

> 🔬 深入 | 前置知识：[LLM 作为推理引擎](./01-llm-reasoning)、[VLM 视觉骨干](./02-vlm-backbone)
> 交叉引用：[模仿学习基础](../01-foundations/04-imitation-learning)、[观测空间与动作空间](../01-foundations/03-observation-action-space)

## 从 VLA 到具身 Agent

VLA 是一个"感知 → 动作"的端到端映射——给定图像和语言指令，直接输出关节控制信号。但现实世界的任务远比单步映射复杂：一个家庭服务机器人需要理解模糊指令、分解子任务、处理意外状况、记住用户偏好、协调多个执行器。

**具身智能 Agent（Embodied Agent）** 在 VLA 的感知-动作回路基础上，增加了推理、记忆、规划、工具使用等高层认知能力，构成一个完整的自主行动系统。如果说 VLA 是机器人的"小脑"（负责运动控制），那么 Agent 框架就是"大脑"（负责决策规划）。

### 时序分析：一个完整推理周期

以"把红色杯子放进洗碗机"任务为例，分析各模块的延迟预算：

| 阶段 | 操作 | 典型延迟 | 频率 |
|------|------|---------|------|
| **感知编码** | RGB×3 → ViT 编码 + 深度处理 | 15-30 ms | 每步 |
| **场景理解** | VLM 检测物体、判断空间关系 | 50-200 ms | 每 1-5 步 |
| **高层规划** | LLM 任务分解（首次 / 重规划时） | 500-2000 ms | 任务开始 / 异常时 |
| **子任务选择** | 当前应执行哪个子任务 | 10-50 ms | 每步 |
| **动作生成** | VLA 生成 Action Chunk（16 步） | 50-150 ms | 每 16 步 |
| **底层执行** | PD 控制器跟踪轨迹 | 1 ms | 1000 Hz |
| **安全检查** | 力矩限制、碰撞检测 | <1 ms | 1000 Hz |

**关键设计约束**：高层规划（~1s）和底层控制（~1ms）之间存在三个数量级的频率差。Agent 框架通过分层架构解决这一矛盾——高层低频决策 + 底层高频控制，中间由 VLA 的 Action Chunking 机制（参见[观测空间与动作空间](../01-foundations/03-observation-action-space)）桥接。

## 记忆架构

记忆是 Agent 区别于无状态 VLA 的核心能力。人类认知科学将记忆分为多种类型，具身 Agent 需要对应实现。

### 三类记忆的详细设计

| 记忆类型 | 认知对应 | 容量 | 实现方式 | 存储内容 | 访问延迟 |
|---------|---------|------|---------|---------|---------|
| **工作记忆** | 短期记忆 | 4K-128K Token | Transformer 上下文窗口 | 当前任务状态、最近几步观测 | <1 ms |
| **情景记忆** | 经历回忆 | 10K-100K 条经验 | RAG + 向量数据库 | 过去的成功/失败经验片段 | 10-50 ms |
| **语义记忆** | 常识知识 | 模型参数级 | 预训练权重 + LoRA | 物体属性、物理规律、操作常识 | <1 ms |
| **技能库** | 程序性记忆 | 100-1000 个技能 | 代码函数 + 参数化策略 | pick(obj), pour(src,dst) 等 | 5-20 ms |

### 情景记忆的 RAG 实现

```
经验存储阶段：
  完成任务 → 提取关键帧 → [CLIP 编码] → 向量 embedding
         → 结构化记录 {任务, 物体, 成功/失败, 策略, 原因}
         → 存入向量数据库 (如 ChromaDB / Pinecone)

经验检索阶段：
  当前观测 → [CLIP 编码] → 查询最相似的 K 条经验
         → 注入 LLM 上下文：
           "过去类似场景中，你用策略 X 成功了，策略 Y 失败了因为..."
         → LLM 据此做出更好的决策
```

**具体示例**：机器人尝试打开一个新品牌的瓶盖。情景记忆检索到之前打开类似瓶子的经验——"圆形瓶盖需要逆时针旋转 → 但上次塑料瓶盖因为太滑失败了 → 改用增大抓取力+缓慢旋转策略后成功"。这些经验被注入当前决策上下文。

### 工作记忆管理

工作记忆的容量有限（即 Transformer 的上下文窗口），必须精心管理：超出容量时压缩旧信息（摘要化后转存到情景记忆），保持最新观测和任务状态在窗口内。关键设计是**信息压缩策略**——将完整视觉观测压缩为文字描述（"桌上有红色杯子在左侧"），减少 Token 占用。

## 工具使用：三个详细场景

具身 Agent 的工具使用（Tool Use）超越了单纯的肢体操作，包括物理工具和数字化工具。

### 场景 1：使用物理工具测量

```
用户指令: "检查这个零件的尺寸是否合格"

Agent 推理过程:
  [感知] 识别出桌面上的金属零件，形状为圆柱体
  [推理] 需要测量直径 → 需要卡尺 → 记忆中知道卡尺在工具架第二格
  [规划] 子任务分解:
    1. 移动到工具架 → 拾取卡尺 (VLA pick 技能)
    2. 移动到零件旁 → 将卡尺夹住零件 (VLA place+grasp 技能)
    3. 读取卡尺刻度 → 调用 VLM 视觉读数
    4. 与规格比较 → LLM 推理：32.1mm vs 标准 32.0±0.2mm → 合格
    5. 放回卡尺 → 向用户报告结果
  [执行] 逐步调用 VLA 底层技能 + VLM 读数 + LLM 判断
```

### 场景 2：调用数字化 API

```
用户指令: "帮我整理厨房，把过期的食品扔掉"

Agent 推理过程:
  [感知] 扫描厨房台面，识别出 5 个食品包装
  [工具调用] 对每个食品:
    → VLM 读取包装上的文字 → 提取保质期日期
    → 调用 date_compare API: is_expired("2025-03-15", today="2026-04-17")
    → 返回: True (已过期)
  [推理] 食品 A、C 已过期；B、D、E 未过期
  [规划] pick-and-place: A → 垃圾桶, C → 垃圾桶
  [执行] VLA 逐一执行抓取-放置
```

### 场景 3：动态异常处理与重规划

```
用户指令: "把咖啡端到客厅桌上"

Agent 执行过程:
  [初始规划] 拿起咖啡杯 → 走到客厅 → 放在桌上
  [执行中] 力传感器检测到端杯时咖啡在晃动
  [异常检测] 液面倾斜超过阈值 → 触发重规划
  [重规划]
    → LLM: "咖啡溢出风险，需要降低移动速度并保持杯子水平"
    → 修改 VLA 输入参数: max_velocity *= 0.3, orientation_constraint = "level"
  [继续执行] 以慢速稳定端送
  [力传感反馈] 正常 → 成功放置
  [记忆存储] 存入经验: "端液体时需要低速+水平约束"
```

## 多 Agent 协作协议

当多个具身 Agent 需要协作完成复杂任务时，需要设计通信协议和任务分配机制。

### 通信架构

```
┌─────────────┐     共享状态黑板       ┌─────────────┐
│  Agent A    │◄────────────────────►│  Agent B    │
│  (搬运臂)   │    ┌──────────────┐   │  (装配臂)   │
│             │    │ 任务状态表    │   │             │
│ 能力:       │    │ 物体位置表    │   │ 能力:       │
│ - 抓取大物体│    │ 协作约束表    │   │ - 精密装配  │
│ - 重载搬运  │    │ 冲突解决队列  │   │ - 螺丝拧紧  │
└─────────────┘    └──────────────┘   └─────────────┘
       ▲                                     ▲
       │         ┌─────────────┐             │
       └────────►│  Agent C    │◄────────────┘
                 │  (质检)      │
                 │             │
                 │ 能力:       │
                 │ - 视觉检测  │
                 │ - 缺陷报告  │
                 └─────────────┘
```

### 任务分配策略

| 策略 | 描述 | 适用场景 | 代表方法 |
|------|------|---------|---------|
| **集中式分配** | 中央调度器统一分配任务 | 工厂流水线，任务可预测 | 线性规划 / 匈牙利算法 |
| **拍卖式分配** | Agent 对任务出价，最优者获得 | 动态仓库分拣 | Contract Net Protocol |
| **LLM 协商** | Agent 间通过自然语言协商 | 开放域家庭任务 | ChatArena / RoCo |
| **层级分配** | 主 Agent 分解任务，子 Agent 执行 | 复杂多步骤任务 | Manager-Worker |

## 代表性 Agent 框架

### SayCan (Google, 2022)

**核心思想**：LLM 提供任务分解（"说什么有用"），value function 评估可行性（"什么做得到"），两者相乘选择动作。

```
任务: "我饿了，想吃点东西"

LLM 规划 (Say):                          可行性评估 (Can):
  1. 找到苹果      → P(有用)=0.8    ×    P(可行)=0.9   = 0.72 ✓ 选择
  2. 找到饮料      → P(有用)=0.3    ×    P(可行)=0.7   = 0.21
  3. 打扫桌子      → P(有用)=0.1    ×    P(可行)=0.8   = 0.08

执行 "找到苹果" → 成功 → 下一步规划...
```

**局限**：需要预定义技能库，技能数量有限（约 500 个），无法处理未见过的任务。

### Voyager (NVIDIA, 2023)

**核心思想**：在 Minecraft 中实现自我驱动的探索 Agent，具有自动课程学习、可执行代码生成、技能库积累三大机制。

| 组件 | 功能 | 实现 |
|------|------|------|
| 自动课程 | 生成探索目标 | GPT-4 根据当前状态提出下一个挑战 |
| 代码生成 | 将计划转化为可执行代码 | GPT-4 生成 JavaScript 控制代码 |
| 技能库 | 积累可复用技能 | 成功代码存入向量数据库，按相似度检索 |
| 自我验证 | 检查任务是否完成 | GPT-4 判断 + 游戏状态检查 |

**关键成果**：在 Minecraft 中获得的技能数量是其他方法的 3.3 倍，能发现钻石（需要 ~40 步规划链）。

### DEPS (Deep Exploration with Plan Selector, 2023)

**核心思想**：解决 LLM 规划中"计划太粗/无法执行"的问题。引入描述器（Descriptor）为 LLM 提供详细环境信息，使计划更可行。

### RoboAgent (CMU, 2023)

**核心思想**：通过少量人类演示（每任务约 5-10 条）+ 大量语义增强（semantic augmentation）实现 12 个技能的泛化。

**与 VLA 的关键区别**：RoboAgent 不使用大语言模型，而是用目标条件化的扩散策略，但它的数据增强策略（环境随机化、纹理替换、光照变化）对 VLA 训练同样有重要参考价值。

## VLA 在 Agent 全栈中的精确定位

```
┌────────────────────────────────────────────────────┐
│ 层级 5: 用户接口层                                    │
│   语音/文本 → 意图理解 → 自然语言任务描述              │
│   代表: Whisper + GPT-4                             │
├────────────────────────────────────────────────────┤
│ 层级 4: 高层推理与规划                                │
│   任务分解 → 子目标序列 → 异常处理 → 重规划            │
│   代表: GPT-4 / Gemini + CoT / ToT                  │
│   频率: ~1 Hz (每秒 1 次高层决策)                     │
├────────────────────────────────────────────────────┤
│ 层级 3: 场景理解与策略选择                             │
│   物体检测 → 空间关系 → 可供性分析 → 技能选择          │
│   代表: PaliGemma / InternVL / CLIP                  │
│   频率: ~5 Hz                                        │
├────────────────────────────────────────────────────┤
│ 层级 2: 运动规划与动作生成  ← VLA 的核心位置            │
│   观测+指令 → 动作序列 (Action Chunk)                 │
│   代表: pi_0 / OpenVLA / RT-2 / Octo                │
│   频率: 5-10 Hz (每 100-200ms 生成一批动作)           │
├────────────────────────────────────────────────────┤
│ 层级 1: 底层运动控制                                  │
│   关节 PD 控制 → 力矩控制 → 安全限位                  │
│   代表: libfranka / ros_control                      │
│   频率: 200-1000 Hz                                  │
├────────────────────────────────────────────────────┤
│ 层级 0: 硬件层                                       │
│   电机驱动 → 传感器采集 → 通信总线                     │
│   代表: EtherCAT / CAN bus                           │
│   频率: 1000+ Hz                                     │
└────────────────────────────────────────────────────┘
```

### 当前的两条技术路线

| 路线 | 描述 | 代表 | 优势 | 劣势 |
|------|------|------|------|------|
| **模块化 Agent** | LLM 规划 + VLA 执行，各层解耦 | SayCan, DEPS, Code as Policies | 可解释性强、模块可替换 | 模块间信息损失、延迟高 |
| **端到端 Agent** | 一个模型完成感知-推理-执行 | RT-2, pi_0-FAST, Gemini Robotics | 信息无损、延迟低 | 可解释性差、需要海量数据 |

**趋势**：两条路线正在融合——端到端模型在内部发展出类似"模块"的隐式结构（如 pi_0 的推理 Token），而模块化系统通过共享表征减少信息损失。

## 局限与失败模式

| 失败模式 | 原因 | 示例 | 缓解策略 |
|---------|------|------|---------|
| **规划幻觉** | LLM 生成不可行的计划 | "飞到二楼拿东西" | SayCan 式可行性过滤 |
| **记忆混淆** | 检索到不相关的经验 | 打开门的经验用于打开抽屉 | 提高 RAG 检索精度 |
| **协作死锁** | 多 Agent 互相等待 | A 等 B 让路、B 等 A 让路 | 超时机制 + 优先级仲裁 |
| **级联故障** | 底层感知错误传播到高层 | 误检物体 → 错误规划 → 危险动作 | 多级置信度过滤 |
| **开放域泛化** | 遇到完全未见过的场景 | 训练中没有"折纸"任务 | Few-shot 学习 + 人类介入 |

## 小结

| 概念 | 要点 |
|------|------|
| 具身 Agent 定义 | VLA + 推理 + 记忆 + 规划 + 工具使用 = 完整自主系统 |
| 感知-推理-行动循环 | 高层规划 ~1Hz、VLA 执行 ~5-10Hz、底层控制 ~1000Hz |
| 记忆架构 | 工作记忆（上下文窗口）+ 情景记忆（RAG）+ 语义记忆（权重）+ 技能库 |
| 工具使用 | 物理工具操作 + 数字 API 调用 + 异常重规划 |
| 多 Agent 协作 | 集中式/拍卖式/LLM 协商/层级式分配 |
| 代表框架 | SayCan（可行性过滤）、Voyager（技能积累）、DEPS（详细描述）、RoboAgent（数据增强） |
| VLA 定位 | Agent 全栈的层级 2（运动规划与动作生成），连接高层推理与底层控制 |
