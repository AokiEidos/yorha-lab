# Helix（Figure AI）详解

> ⚙️ 进阶 → 🔬 深入 | 前置知识：[VLA 架构对比表](../02-architecture/05-architecture-comparison)，[π₀ 详解](./06-pi-zero)

## 定位

**Helix**（Figure AI, 2025）是 Figure AI 为其人形机器人 Figure 02 开发的 VLA 系统。核心创新在于**双系统架构（Dual-System Architecture）**——灵感来自认知科学中 Daniel Kahneman 的"快思考"与"慢思考"理论（System 1 / System 2），将高层语义推理与低层运动控制彻底解耦。

Helix 并非仅仅是一个研究原型——它已经在 BMW 南卡罗来纳州斯帕坦堡工厂进行试点部署，是全球最早进入真实制造业产线的人形机器人 VLA 系统之一。

## 为什么人形机器人需要双系统

在深入架构之前，需要理解一个根本性约束：

**人形机器人的双足行走需要极高频率的平衡控制。** 人类的脊髓反射弧约在 50-200ms 内完成一次平衡修正，相当于 5-20 Hz。工程上，为了保证双足步态的稳定性，关节级控制器通常运行在 **200-1000 Hz**。

而当前最快的 VLM（如 PaliGemma 3B）在 GPU 上的推理延迟约为 200-500ms，即 **2-5 Hz**。更大的 VLM（如 70B 级别）延迟在 1-2 秒。

```
控制需求的频率谱：

1000 Hz ┤ ███ 关节伺服 PD 控制
 200 Hz ┤ ███ 全身平衡/步态控制 ← System 1 必须覆盖
  50 Hz ┤ ███ 手臂轨迹跟踪
  10 Hz ┤ ░░░ 抓取规划刷新
   2 Hz ┤ ░░░ VLM 场景推理 ← System 2 的实际频率
   1 Hz ┤ ░░░ 高层任务规划
```

**如果用单一 VLM 控制全身——推理一次 500ms，这期间机器人早就摔倒了。** 这是双系统架构的根本动机：将"思考什么做"和"怎么不摔倒地做"解耦到两个频率截然不同的子系统。

### System 2（慢系统 / "大脑"）

System 2 是 Helix 的认知中枢，负责"理解世界"和"决定做什么"。

| 属性 | 规格 |
|------|------|
| **模型类型** | 大型 VLM（具体架构未完全公开，推测 10B+ 参数量级） |
| **输入** | 多相机 RGB 图像 + 语言指令 + 任务上下文 |
| **输出** | 高层目标向量、子任务指令序列、语义引导信号 |
| **推理频率** | ~1-2 Hz（每次推理 500ms-1s） |
| **运行硬件** | Figure 02 机载 GPU 或边缘云计算 |

**System 2 的典型推理过程：**

```
输入: 相机图像 + 指令 "请把桌上的零件放到传送带上"

Step 1 - 场景理解:
  → 识别桌面上有 3 个金属零件、1 条传送带
  → 估计零件位置 (x, y, z) 和朝向
  → 判断传送带运行方向和速度

Step 2 - 任务推理:
  → 分解为子任务: [接近桌面] → [抓取零件A] → [移动到传送带上方] → [放置]
  → 选择抓取策略: 侧面夹取 vs 顶部抓取

Step 3 - 目标生成:
  → 输出: 目标空间坐标 (0.35, -0.12, 0.85)
  → 输出: 抓取姿态四元数 (0.707, 0, 0.707, 0)
  → 输出: 力控制阈值 15N
```

### System 1（快系统 / "脊髓"）

System 1 是 Helix 的运动执行器，负责"安全高效地执行动作"。

| 属性 | 规格 |
|------|------|
| **模型类型** | 轻量级策略网络（推测为小型 Transformer 或 MLP 集成，参数量 < 100M） |
| **输入** | 低延迟视觉特征 + 本体感觉（关节角度/角速度/力矩） + System 2 目标 |
| **输出** | 全身 40+ 关节的力矩/位置命令 |
| **控制频率** | ~200 Hz（5ms 一个控制周期） |
| **延迟** | < 5ms 端到端推理 |

**System 1 必须同时处理多个层次的控制：**

```
Layer 1: 全身平衡控制 (最高优先级)
  → 基于 IMU + 足底压力传感器
  → ZMP (零力矩点) 保持在支撑多边形内
  → 频率: 200-500 Hz

Layer 2: 步态生成与跟踪
  → 基于 CPG (中央模式发生器) 或学习策略
  → 生成双足行走/转向/停止模式
  → 频率: 100-200 Hz

Layer 3: 上肢操作执行
  → 跟踪 System 2 给出的末端执行器目标
  → 全身逆运动学 + 阻抗控制
  → 频率: 50-100 Hz
```

### 系统间通信协议

两个系统之间的通信是 Helix 架构的关键设计。核心挑战：**如何让一个 1 Hz 的系统有效驱动一个 200 Hz 的系统？**

```
System 2 (1-2 Hz)                    System 1 (200 Hz)
    │                                     │
    │── 目标向量 G_t ──────────────────→  │ 接收 G_t
    │   (空间坐标+姿态+约束)              │ 在 G_t 和 G_{t+1} 之间
    │                                     │ 自主插值 200 个控制步
    │                                     │ 同时维持平衡
    │                                     │
    │   (等待 ~500ms-1s)                  │ ... 执行中 ...
    │                                     │
    │── 目标向量 G_{t+1} ─────────────→  │ 平滑切换到新目标
    │                                     │
    │← 状态反馈 S_t ──────────────────── │ 返回执行状态
    │   (完成度/异常/接触力)              │
```

**目标向量 G_t 的可能格式（推测）：**

```python
# 伪代码：System 2 → System 1 通信结构
class GoalVector:
    ee_target_pos: np.ndarray    # 末端执行器目标位置 (3,)
    ee_target_quat: np.ndarray   # 末端执行器目标姿态 (4,)
    gripper_state: float         # 夹爪开合度 [0, 1]
    gaze_target: np.ndarray      # 视线目标位置 (3,)
    locomotion_vel: np.ndarray   # 期望移动速度 (vx, vy, vθ)
    force_threshold: float       # 力控制阈值
    subtask_embedding: np.ndarray # 子任务语义嵌入 (D,)
    priority_mask: np.ndarray    # 各层控制优先级权重
```

### 安全机制

双系统架构为安全性提供了天然优势：

```
安全层级:

Level 0: 硬件安全 (不依赖 AI)
  → 关节力矩硬限制
  → 电流过载保护
  → 物理紧急停止按钮

Level 1: System 1 实时安全 (200 Hz)
  → 碰撞检测与反应 (< 10ms)
  → 关节极限软约束
  → 摔倒保护姿态
  → 力控制上界

Level 2: System 2 语义安全 (1-2 Hz)
  → 危险动作识别与拒绝
  → 人类检测与安全距离维护
  → 任务合理性验证

关键特性: 即使 System 2 完全失效，System 1 仍可独立
维持平衡并安全停止 → 故障降级（Graceful Degradation）
```

## 与 GR00T N1 双系统的对比

Helix 和 GR00T N1 都采用双系统架构，但设计哲学有显著差异：

| 维度 | Helix (Figure AI) | GR00T N1 (NVIDIA) |
|------|-------------------|-------------------|
| **System 1 架构** | 轻量策略网络（MLP/小Transformer） | Diffusion Transformer |
| **System 1 生成方式** | 确定性前向传播 | 扩散去噪（多步采样） |
| **System 1 延迟** | ~5ms（估计） | ~10-50ms（扩散需多步） |
| **System 2 架构** | 大型 VLM（自研） | 大型 VLM（NVIDIA 生态） |
| **训练数据来源** | 真机遥操作 + 有限仿真 | Isaac Sim 大规模仿真为主 |
| **仿真依赖** | 低（真机优先策略） | 高（仿真生态深度绑定） |
| **硬件绑定** | Figure 02 专用 | 跨具身体通用设计 |
| **商业阶段** | 工厂试点部署中 | 平台/生态阶段 |
| **安全机制** | 双层解耦 + 硬件安全 | 仿真验证 + 策略约束 |

**设计哲学差异：**

- **Helix** 倾向"垂直整合"——自研硬件、自研模型、自采数据，在特定场景（工厂）做到极致可靠
- **GR00T N1** 倾向"平台生态"——提供通用框架和仿真工具，让合作伙伴在各种人形机器人上使用

## 真实部署案例：BMW 工厂

Figure AI 与 BMW 的合作是人形机器人 VLA 最重要的工业验证之一：

**部署场景**：BMW 斯帕坦堡工厂（Spartanburg, South Carolina），年产超 40 万辆汽车

**已验证任务**：
- 将钣金零件从料箱放置到定位夹具上
- 在传送带旁进行零件质量目视检查
- 轮毂等重复性搬运任务

**关键性能指标（推测范围）**：

| 指标 | 数值 |
|------|------|
| 单任务周期时间 | 8-15 秒/次 |
| 成功率（受控场景） | > 95% |
| 连续运行时间 | 4+ 小时/充电 |
| 位置精度 | ±2-5 mm |
| 安全停止响应时间 | < 100ms |

**为什么选择人形机器人而非传统工业臂？**
1. 工厂产线为人设计——工位尺寸、工具高度都适配人体
2. 人形机器人可以直接进入现有产线，无需改造基础设施
3. 通用性——同一台机器人可以适应不同工位

## 训练方法

Helix 的训练流程（基于公开信息推断）：

```
Stage 1: VLM 预训练 (System 2)
  → 在大规模互联网图文数据上预训练视觉-语言理解
  → 数据量: 数十亿图文对
  → 获得: 场景理解、常识推理、语言理解能力

Stage 2: 策略网络预训练 (System 1)
  → 在仿真环境中训练基础运动技能
  → 包括: 行走、平衡、基础抓取
  → 方法: 强化学习 + 模仿学习混合

Stage 3: 端到端联合微调
  → 真机遥操作数据（人类操作员穿戴动捕服控制 Figure 02）
  → 数据量: 推测数十万条轨迹
  → System 1 和 System 2 联合优化通信接口

Stage 4: 工厂场景专项微调
  → 针对 BMW 工厂的具体任务和环境
  → 小规模真机数据 (~数百到数千条轨迹)
  → 重点优化成功率和周期时间
```

## 技术挑战与局限

| 挑战 | 现状 |
|------|------|
| **System 间延迟** | 目标更新在两次推理间有 500ms-1s 的"盲区" |
| **故障传播** | System 2 给出错误目标时，System 1 会忠实执行错误动作 |
| **闭源生态** | 架构细节未公开，社区无法复现和改进 |
| **场景泛化** | 当前主要在受控工厂环境验证，家庭等非结构化环境未知 |
| **成本** | Figure 02 单台成本推测 > $50,000，规模化降本路径待验证 |

## 小结

| 概念 | 要点 |
|------|------|
| **双系统架构** | System 2（VLM ~1Hz 推理）+ System 1（策略网络 ~200Hz 控制） |
| **设计动机** | 人形双足平衡需要 200Hz，VLM 推理只有 1-2Hz，频率矛盾必须解耦 |
| **System 间通信** | 目标向量传递 + 状态反馈，System 1 在目标间自主插值 |
| **安全机制** | 三级安全（硬件/System 1 实时/System 2 语义），支持故障降级 |
| **硬件平台** | Figure 02：40+ DoF、灵巧手、170cm 人形 |
| **工业验证** | BMW 斯帕坦堡工厂试点部署，零件搬运/放置任务 |
| **vs GR00T N1** | Helix 垂直整合真机优先，GR00T N1 平台生态仿真优先 |
