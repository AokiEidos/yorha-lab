## 四种设计模式

### 模式 A：VLM 直出（端到端最简）

```
┌─────────────────────────────────────┐
│           VLM (单一模型)              │
│                                     │
│  [视觉编码器]──→                     │
│                 ├→[LLM]→ 动作 Token  │
│  [文本编码器]──→                     │
│                                     │
│  一个模型做一切: 理解+推理+动作       │
└─────────────────────────────────────┘

代表：RT-2, OpenVLA
参数量：7-55B
控制频率：1-10 Hz
优势：架构最简，端到端优化，利用预训练知识
劣势：离散化精度损失，单峰预测，频率受限于模型大小

适用场景:
  ✅ 需要语言指令泛化
  ✅ 快速原型验证（OpenVLA 微调即用）
  ✅ 推理密集型任务（语义推理）
  ❌ 灵巧操作（精度不足）
  ❌ 高频控制需求（>10 Hz）
```

### 模式 B：VLM + 独立解码头

```
┌─────────────────────────────────────┐
│                                     │
│  [VLM 编码器] → 语义特征             │
│       │                             │
│       ▼                             │
│  [动作解码头] → 连续动作序列          │
│   Diffusion / Flow Matching / MLP   │
│                                     │
│  理解和动作生成职责分离               │
└─────────────────────────────────────┘

代表：π₀ (Flow Matching), Octo (Diffusion/MLP)
参数量：93M - 3B
控制频率：10-50 Hz
优势：连续动作高精度，多模态动作分布，可独立优化解码头
劣势：架构复杂度增加，两模块协调训练

适用场景:
  ✅ 灵巧操作（折衣/装配/烹饪）
  ✅ 需要多模态动作建模
  ✅ 高频控制（π₀ 实现 50 Hz）
  ✅ 跨具身体（不同动作维度用不同解码头）
  ❌ 推理密集型任务（VLM 部分通常较小）
```

### 模式 C：双系统架构（人形机器人前沿）

```
┌─────────────────────────────────────┐
│  System 2 (慢, ~1-5 Hz)             │
│  [VLM] → 场景理解/任务规划           │
│       │                             │
│       ▼ 目标向量/子任务               │
│                                     │
│  System 1 (快, ~30-200 Hz)           │
│  [快速策略网络] → 低延迟关节命令      │
│  实时平衡 + 动作执行                  │
└─────────────────────────────────────┘

代表：Helix (MLP System 1), GR00T N1 (DiT System 1)
参数量：System 1 ~100-300M, System 2 ~10B+
控制频率：30-200 Hz (System 1)
优势：解耦推理和控制，满足高频平衡需求，支持故障降级
劣势：系统间通信设计复杂，目标更新有延迟盲区

适用场景:
  ✅ 人形机器人全身控制
  ✅ 需要实时平衡的双足行走
  ✅ 大规模工业部署（需要安全层级）
  ❌ 简单桌面操作（过度设计）
  ❌ 资源受限场景
```

### 模式 D：无 LLM 的轻量级方案

```
┌─────────────────────────────────────┐
│                                     │
│  [视觉编码器] → 特征                 │
│       │                             │
│       ▼                             │
│  [Transformer / CVAE] → 动作         │
│                                     │
│  无 LLM 参与，纯视觉运动策略          │
└─────────────────────────────────────┘

代表：ACT, Diffusion Policy, Octo (小配置), GR-1
参数量：35M - 130M
控制频率：20-100 Hz
优势：极轻量，延迟极低，单任务性能可以很高
劣势：不理解语言指令（或理解有限），泛化能力弱

适用场景:
  ✅ 资源受限（边缘设备，Jetson Nano）
  ✅ 单任务高性能（一个任务优化到极致）
  ✅ 高频控制需求
  ✅ 无需语言指令的固定任务
  ❌ 需要泛化到新任务/物体
  ❌ 需要自然语言交互
```

## 架构选择决策树

```
你要做什么？
│
├── 桌面操作（单臂/双臂）
│   │
│   ├── 需要语言指令泛化？
│   │   ├── 是 → 需要灵巧操作（折叠/装配）？
│   │   │       ├── 是 → 模式 B (π₀)
│   │   │       └── 否 → 模式 A (OpenVLA, 开源快速上手)
│   │   └── 否 → 单任务高性能？
│   │           ├── 是 → 模式 D (ACT, 50-100 条数据即可)
│   │           └── 否 → 模式 D (Octo, 小配置)
│   │
│   └── 预算和硬件？
│       ├── 1× RTX 3090/4090 → OpenVLA / ACT / Octo
│       ├── 1× A100 → π₀ / OpenVLA
│       └── 多卡/云端 → RT-2 级别
│
├── 人形机器人
│   │
│   ├── 需要双足行走？
│   │   └── 是 → 模式 C (Helix/GR00T N1 风格双系统)
│   │
│   └── 仅上半身操作？
│       └── 模式 B (π₀ 风格) 或 模式 C 简化版
│
├── 移动操作（底盘+臂）
│   └── 模式 A 或 B + 导航模块
│
├── 四足/非标准具身体
│   └── 模式 B + 自定义动作空间 (参考 QUAR-VLA)
│
└── 最强泛化+推理（不计成本）
    └── 模式 A 扩展版 (Gemini Robotics 级别)
```

## 具体部署场景推荐

### 场景 1：学术研究 / 快速原型

```
推荐: OpenVLA (7B) + LoRA 微调
理由:
  - 开源，社区活跃，文档完善
  - 单卡 A100 可训练和部署
  - 200-500 条数据即可微调到特定任务
  - 作为 baseline 与自己的方法对比

硬件: 1× A100 (40GB) 或 2× RTX 4090 (24GB)
数据: 200-500 条遥操作轨迹
训练: 6-12 小时 (LoRA, rank 64)
推理: ~150ms (~7 Hz)
```

### 场景 2：工业桌面操作（分拣/装配）

```
推荐: ACT (ALOHA) 或 π₀
理由:
  - ACT: 50-100 条演示数据即可，极快部署
  - π₀: 灵巧操作更强，跨任务泛化更好
  - 工业场景任务相对固定，不需要强语言理解

硬件: ACT → 1× RTX 3090; π₀ → 1× A100
数据: ACT → 50-100 条/任务; π₀ → 数百到数千条
训练: ACT → 数小时; π₀ → 1-3 天
推理: ACT → ~10ms (~100 Hz); π₀ → ~20ms (~50 Hz)
```

### 场景 3：家庭服务机器人

```
推荐: 模式 A (OpenVLA/未来开源大 VLA) + 导航模块
理由:
  - 需要理解多样化的自然语言指令
  - 需要泛化到家庭中各种物体和场景
  - 任务不需要极高精度（搬运、整理为主）
  - 语义理解比灵巧操作更重要

硬件: 机载 Jetson + 可选云端推理
数据: 大规模预训练 + 家庭场景微调
推理: ~5-10 Hz (足够搬运任务)
```

### 场景 4：人形机器人（工厂部署）

```
推荐: 模式 C 双系统 (Helix/GR00T N1 风格)
理由:
  - 双足行走需要 >100 Hz 的平衡控制
  - 工厂任务需要足够的智能规划
  - 安全性要求极高（与人共存）
  - 需要长时间稳定运行

硬件: Jetson Thor 或同级别 SoC
System 1: ~200 Hz, ~5ms 延迟
System 2: ~2 Hz, ~500ms 延迟
安全: 三级安全架构 (硬件/S1实时/S2语义)
```

## 参数效率分析

不同架构的参数效率（性能 / 参数量）差异显著：

```
参数效率排名 (粗略):

高效率                                     低效率
←────────────────────────────────────────────→
ACT     Octo    π₀     OpenVLA   RT-2    Gemini
(80M)   (93M)   (3B)   (7.6B)   (55B)   (>100B)
│       │       │       │        │       │
│  单任务│  通用  │ 灵巧  │  通用   │ 涌现  │ 最强泛化
│  极高效│  高效  │ 高效  │  中等   │ 有效  │ 低效率
│       │       │       │        │       │ (但能力最强)

关键观察:
1. 轻量模型 (ACT, 80M) 在单任务上效率极高
2. π₀ (3B) 是效率和能力的最佳平衡点
3. 超大模型 (>55B) 参数效率低，但涌现能力无法替代
4. 对大多数应用，3-7B 是性价比甜点区间
```

## 未来架构趋势预测

**2023-2024 路线探索期**：多条路线并行验证（离散 Token vs 连续动作，单系统 vs 双系统）。**2025-2026 路线收敛期**（当前）：双系统成为人形标配，Flow Matching 成为动作头主流，3-7B VLM backbone 成为社区标准。**2027+ 融合成熟期**（预测）：大模型推理 (System 2) + 专用动作头 (System 1) 统一架构，模型压缩使 100B 级可在边缘运行。

## 小结

| 概念 | 要点 |
|------|------|
| **四种模式** | A=VLM 直出(简)、B=VLM+解码头(精)、C=双系统(人形)、D=无LLM(轻) |
| **性能甜点** | 3-7B 参数区间是性价比最优，π₀(3B) 是灵巧操作标杆 |
| **频率范围** | 1Hz(Gemini) → 10Hz(OpenVLA) → 50Hz(π₀) → 200Hz(Helix S1) |
| **开源选择** | 快速原型=OpenVLA，单任务=ACT，通用=Octo |
| **人形必选** | 双系统架构（模式C），System 1 ≥30Hz 是硬性要求 |
| **成本考量** | ACT $10 起步，OpenVLA LoRA $100 级别，工业方案 $10K+ |
