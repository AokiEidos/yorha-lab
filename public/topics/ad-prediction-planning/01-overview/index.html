01 自动驾驶预测与规划算法全景

基于感知-预测-规划-控制的完整技术栈

从传感器到执行：完整技术栈 / Full Stack from Sensor to Actuation

自动驾驶系统的核心任务是让车辆在没有人类干预的情况下安全、高效地从 A 点到达 B 点。这个看似简单的目标，背后需要一整套复杂的技术栈协同工作。

自动驾驶全栈技术架构

各层的职责边界 / Layer Responsibilities

感知层 Perception

感知层的任务是"看懂世界"。从原始传感器数据中提取结构化信息——哪里有车、行人、障碍物，车道线在哪里，交通信号灯是什么状态。

核心技术是 BEV 感知（Bird's Eye View Perception），即将多个摄像头的 2D 图像投影到统一的鸟瞰视角表示。

代表工作BEVFormer（Li et al., 2022）使用时空 Transformer 在 BEV 空间做 3D 检测和分割，将多摄像头信息统一到自车坐标系下。

预测层 Prediction

预测层的任务是"预测未来"。给定当前所有交通参与者的历史轨迹和地图信息，预测每个参与者在未来 5-8 秒内的运动轨迹。

这是一个天然的多模态问题（Multi-modal Problem）——同一个路口，前方车辆可能直行、左转或右转，每种都是合理的。预测层需要输出轨迹的概率分布，而非单一预测。

规划层 Planning

规划层的任务是"决定怎么走"。基于感知结果和预测分布，为自车规划一条安全、高效、舒适的行驶轨迹。

规划分三个层级：

  • 路由规划（Route Planning）：全局地图上找最优路径（类似导航）
  • 行为规划（Behavior Planning）：决定高层策略（变道、让行、超车）
  • 运动规划（Motion Planning）：生成具体轨迹点（位置、速度、加速度）

控制层 Control

控制层的任务是"精确执行"。将规划层输出的期望轨迹转化为车辆底层的油门、刹车和转向角指令。

常用方法：

  • PID 控制（Proportional-Integral-Derivative）：简单跟踪
  • MPC（Model Predictive Control）：考虑车辆动力学约束的最优跟踪
  • LQR（Linear Quadratic Regulator）：线性二次调节

感知 / 预测 / 规划 / 控制：四大模块职责 / Four Module Responsibilities

模块化方案将自动驾驶系统拆解为独立模块，每个模块有明确的输入输出接口。

模块化 vs 端到端：两大范式 / Modular vs End-to-End

这是自动驾驶算法架构的核心 trade-off，两种范式各有优劣。

优点：

  1. 可解释性强：每个模块有明确的输入输出接口，出问题可以定位到具体模块
  2. 独立评估：感知精度（mAP）、预测精度（minADE）、规划安全性（碰撞率）分别衡量
  3. 团队协作：感知团队、预测团队、规划团队可以并行开发
  4. 安全验证：每个模块可以单独做单元测试和安全验证
  5. 增量升级：可以单独升级某个模块而不影响其他模块

缺点：

  1. 信息瓶颈（Information Bottleneck）：感知→预测之间只传递检测框，丢失大量原始传感器信息
  2. 误差级联（Error Cascading）：感知漏检直接导致预测和规划失败，误差逐级放大
  3. 目标不对齐：感知优化 mAP 但规划需要"对规划有影响的检测"
  4. 手工接口设计：模块间接口是人为设计的，未必是信息传递最优方式

端到端方案 End-to-End Approach

传感器
→
统一神经网络
→
规划 / 控制

感知特征 → 预测特征 → 规划特征 → 控制信号，端到端梯度反传

优点：

  1. 无信息损失：所有信息在统一特征空间中流转，不需要瓶颈接口
  2. 目标一致：整个系统端到端优化最终驾驶质量指标
  3. 自动学习表示：模型自动学习对规划最有用的中间表示
  4. 梯度直通：规划需求可以反传到感知层

缺点：

  1. 不可解释：中间表示不再有明确语义，出了事故难以定位原因
  2. 难以验证：无法对子模块做单元测试
  3. 数据饥渴：需要大量带规划标签的驾驶数据
  4. 监管挑战：向监管机构证明黑箱系统安全性极其困难
  5. 灾难性失败：一旦失败整个系统同时失效，没有 fallback

工业界的务实选择大多数公司目前采用"模块化主架构 + 端到端辅助"的混合方案（Hybrid Approach）。例如 UniAD（Li et al., 2023）保留了感知→预测→规划的模块化结构，但允许端到端梯度流通；工业部署中通常在端到端模型之上叠加一个传统的安全校验层（Safety Checker）作为兜底。

核心挑战 / Core Challenges

自动驾驶从研究到落地面临四大核心挑战。

挑战一：长尾场景 Long-tail Scenarios

自动驾驶最大的技术挑战不是"99% 的常见场景"，而是"1% 的长尾场景"。一个系统可能在 99.9% 的时间里表现完美，但那 0.1% 的失败——异形载具、倒地行人、未标注的施工区域——就可能导致严重事故。

数据层面：长尾场景在训练数据中出现频率极低（可能百万帧才一次），传统均匀采样训练无法充分学习。解决思路包括：重要性采样（oversampling rare cases）、数据增强（augmentation）、仿真生成（simulation）、世界模型（World Model）生成。

评估层面：离线 benchmark 的测试集也受长尾偏斜影响，一个模型可能 minADE 指标很好但在罕见场景中崩溃。这也是为什么闭环评估（closed-loop evaluation）越来越受重视。

挑战二：实时性 Real-time Constraints

自动驾驶系统需要在严格的时间约束下运行。从传感器采集到控制执行，整个流水线的总延迟通常需要控制在 100-200ms 以内。

模块典型延迟预算主要瓶颈
感知50-80msBEV 特征提取、3D 检测
预测20-50ms多智能体交互建模
规划20-50ms轨迹搜索 / 优化
控制5-10ms通常不是瓶颈
Diffusion 模型的实时性挑战标准 Diffusion 推理需要 50-100 步去噪，单次推理耗时 100-500ms，对预测和规划层都超出了延迟预算。加速方向包括 DDIM 少步采样、一致性蒸馏、TensorRT 部署优化等。

挑战三：安全性 Safety

自动驾驶的安全性要求远超其他 AI 应用。航空领域的安全标准是每飞行小时 $10^{-9}$ 级的灾难性失效率。自动驾驶尚未有统一标准，但业界普遍认为至少需要比人类驾驶安全 10 倍以上才能被公众接受。

概率模型的安全困境深度学习模型本质上输出概率分布，无法提供硬安全保证。即使模型在 99.99% 的情况下输出安全轨迹，剩下的 0.01%（每百万次决策中 100 次不安全）对自动驾驶而言仍不可接受。这催生了混合安全架构——用学习模型做提议，用传统方法做安全校验。

挑战四：数据稀缺与标注成本 Data Scarcity and Annotation Cost

预测和规划相比感知的额外困难在于：轨迹标注需要跟踪时间序列（不只是标注单帧），规划标注需要人类专家的驾驶决策（而非简单的框标注）。这也是为什么自监督学习和仿真数据在预测规划领域越来越重要。

数据类型获取难度标注成本规模现状
2D 图像标注（检测框）易~$0.5/帧数亿帧
3D 点云标注（3D 框）中~$5/帧数千万帧
轨迹标注（未来运动）中~$2/轨迹数百万条

长尾场景标注极难需要合成极少
