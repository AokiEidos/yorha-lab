第一阶段：模块化流水线（2015-2020）

架构

经典模块化架构

LiDAR/Camera → BEV 感知 → 目标跟踪 → 轨迹预测 → 行为规划 → 运动规划 → PID/MPC
输出
  • BEV 感知 → 3D 检测框 + 语义分割
  • 目标跟踪 → 跟踪 ID + 历史轨迹
  • 轨迹预测 → K 条候选轨迹/概率
  • 运动规划 → 轨迹点 (x,y,t,v)
特点
  • 每模块独立开发、评估、迭代
  • 总延迟: ~200ms (串行累加)
  • 接口: 结构化数据

为什么模块化统治了 5 年

  1. 工程可行性：每个模块可以由不同团队独立开发
  2. 可调试性：出问题可以定位到具体模块（是感知漏检还是预测偏差）
  3. 安全验证：每个模块可以单独做单元测试
  4. 数据效率：感知只需要 3D 标注，预测只需要轨迹标注，不需要端到端的驾驶标签

模块化的根本局限

信息瓶颈（Information Bottleneck）：模块间接口只传递结构化数据（检测框、轨迹），丢失了大量信息：

感知 → 预测的信息损失

感知阶段拥有的信息
  • 每个像素的特征 (数百万维)
  • 遮挡物体的部分线索
  • 远处物体的不确定性分布
  • 场景的全局上下文
传递给预测的信息
  • 检测框 (位置+大小+类别, ~10 维/物体)
  • 遮挡物体? 丢失 ✗
  • 不确定性? 丢失 ✗
  • 全局上下文? 丢失 ✗
信息保留率: <1%

第二阶段：感知-预测联合（2020-2022）

核心思想：让感知和预测共享 BEV 特征，减少信息瓶颈。

代表工作

FaF（Fast and Furious, Luo et al., CVPR 2018）：首个联合检测+预测的模型，从 LiDAR BEV 同时输出 3D 检测框和未来运动。

MotionNet（Wu et al., CVPR 2020）：在 BEV 空间直接预测每个网格的未来占据和运动场（motion field），不需要显式的目标检测。

感知-预测联合

LiDAR → BEV 编码器 → BEV 特征 F
┌─────────┼─────────┼─────────┐
检测头 预测头 分割头
3D 框│未来轨迹│语义地图
共享 BEV 特征，不再有信息瓶颈

改进：预测模块可以直接访问感知的 BEV 特征，不再受限于检测框接口。

残留问题：规划模块仍然独立——它接收的还是结构化的轨迹，无法利用 BEV 特征。

第三阶段：全栈联合 UniAD/VAD（2023）

UniAD：统一自动驾驶

UniAD（Li et al., CVPR 2023, Best Paper） 是端到端自动驾驶的里程碑工作。它将感知→预测→规划统一在一个 Transformer 框架中，通过 Query 传递机制 实现模块间的信息流通，同时保留了每个模块的可解释性。

UniAD 核心架构

Camera → BEV 编码器 → BEV 特征
│
┌───────────┼───────────┐
TrackFormer 目标跟踪 │ MapFormer 地图分割 │ MotionFormer 运动预测 │ OccFormer 占据预测
│ Query 传递 │
PlanningFormer 自车规划
↓
输出: 规划轨迹 (L2 距离 + 碰撞率)

关键创新：

  1. Query 传递：Track Query → Motion Query → Planning Query，信息在模块间通过 Query 流转
  2. 端到端梯度：规划的 loss 可以反传到感知层
  3. 保留模块结构：虽然端到端训练，但每个模块仍有独立的 loss 和评估指标

VAD（Vectorized Scene Representation for AD）

VAD（Jiang et al., ICCV 2023） 与 UniAD 同期，采用向量化场景表示：

UniAD
BEV 特征 → Query Transformer → 规划
✓ 多任务联合训练
优点: 信息丰富
缺点: 计算量大
VAD
BEV → 向量化场景 → 规划
✓ 向量化场景表示
优点: 更轻量紧凑
缺点: 压缩可能丢失细节

为什么 UniAD 能成为 CVPR Best Paper

  1. 首次证明了全栈端到端的可行性：之前业界普遍认为端到端不可控、不可调试
  2. 性能提升显著：在 nuScenes 规划 benchmark 上，UniAD 的碰撞率比模块化方案低 50%
  3. 保留了可解释性：Query 传递机制让每个模块的输出仍可可视化和评估
  4. 优雅的架构设计：统一的 Transformer Query 机制，简洁且有效

为什么花了 5 年：三重障碍

从 2018 年首次提出端到端自动驾驶概念，到 2023 年 UniAD 真正实现全栈联合训练，为什么花了 5 年？

障碍一：数据

需求模块化端到端
感知标注3D 框（已有数百万帧）需要
轨迹标注跟踪序列需要
规划标签不需要需要（专家驾驶记录）
数据规模要求中等巨大（长尾场景覆盖）

端到端模型需要"驾驶标签"——即人类专家在每个场景中的驾驶决策。这比画 3D 框困难得多，且长尾场景的覆盖需要海量数据。

障碍二：算力

规划1 × A100, 0.5 天(包含在端到端中)
总计~6 GPU·天~56 GPU·天 (9×)

这三重障碍在 2023 年前同时存在，是端到端方案迟迟无法落地的根本原因。UniAD 的突破在于：(1) nuScenes 提供了足够的数据和 benchmark，(2) A100 提供了足够的算力，(3) Query 传递机制在端到端和可解释性之间找到了平衡点。

UniAD 的核心设计哲学

UniAD（Unified Autonomous Driving, Li et al., CVPR 2023 Best Paper） 的核心思想是：用统一的 Query-based Transformer 架构将感知、跟踪、地图、运动预测和规划串联在一起，实现端到端训练的同时保留每个子模块的可解释性。

关键洞察：模块化方案的问题不在于"有模块"，而在于"模块间只传递结构化数据"。UniAD 保留了模块结构，但将模块间的接口从"检测框/轨迹"升级为"Query 特征"——高维、可微分、信息丰富。

BEV 编码器

UniAD 使用类似 BEVFormer 的架构将多摄像头 2D 图像转换为统一的 BEV 特征。

BEV 编码器 (类似 BEVFormer)

6 摄像头 → ResNet-101/Swin-T → 多尺度 2D 特征
↓
BEV Query (200×200 网格)
↓
Spatial Cross-Attention
  • BEV Query 反投影到 6 摄像头
  • 在 2D 特征图上采样
  • 融合为 BEV 特征
Temporal Self-Attention
  • 当前帧 + 历史帧 BEV 特征

Query 传递机制

UniAD 的核心创新是 Query 传递（Query Propagation）——模块间不传递结构化数据，而是传递 Query 特征。

传统模块化
感知 → 预测
检测框: (x,y,z,w,h,l,θ)
7 个数值/物体
信息量: 极少 ✗
UniAD Query 传递
TrackFormer → MotionFormer
Q_track ∈ R^{N×256}
256 维特征/物体
信息量: 丰富 ✓
外观特征、运动模式、类别概率、注意力权重...
且可微分! → 端到端梯度

Query 依赖关系图

Query 依赖关系图

依赖关系
  • Q_track: 依赖 F_bev
  • Q_map: 依赖 F_bev
  • Q_mot: 依赖 Q_track + Q_map + F_bev
  • Q_ego: 依赖 Q_mot + Q_map + F_bev

每个 Query 都"站在前人的肩膀上"——Motion Query 使用 Track Query 和 Map Query 的信息，Ego Query 使用所有前序 Query 的信息。这种设计让每个模块都能访问丰富的上下文，同时保持清晰的模块边界。

端到端梯度流分析

UniAD 的局限与后续发展

  1. 仅限摄像头输入：原始 UniAD 不支持 LiDAR 融合。后续 UniAD++ 加入了 LiDAR
  2. 规划能力有限：规划头是简单的 MLP，不支持多模态规划
  3. 推理速度：~300ms/帧，远不满足实时要求
  4. 闭环评估缺失：nuScenes 的规划评估是离线的，不反映真实驾驶
